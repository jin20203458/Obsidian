접근 통제는 허가된 사용자, 프로세스, 시스템만이 특정 시스템이나 데이터에 접근할 수 있도록 제어하는 보안 기술입니다. 대표적인 접근 통제 기법으로는 **임의적 접근 통제(DAC)**, **강제적 접근 통제(MAC)**, **역할 기반 접근 통제(RBAC)**가 있습니다.[star7sss.tistory+2](https://star7sss.tistory.com/879)​

## 1. 임의적 접근 통제 (DAC, Discretionary Access Control)

데이터의 소유자(Owner)가 자신의 판단에 따라 다른 사용자에게 접근 권한을 부여하거나 변경할 수 있는 방식입니다.[swingswing.tistory+1](https://swingswing.tistory.com/43)​

- **통제 주체**: 데이터 소유자 (사용자)
    
- **특징**:
    
    - 객체를 생성한 사용자가 해당 객체에 대한 모든 권한을 가지며, 이 권한을 다른 사용자에게 부여할 수 있습니다.[twilstudio](https://twilstudio.com/entry/%EC%A0%95%EB%B3%B4%EC%B2%98%EB%A6%AC%EA%B8%B0%EC%82%AC-%EB%8D%B0%EC%9D%B4%ED%84%B0%EB%B2%A0%EC%9D%B4%EC%8A%A4-%EB%B3%B4%EC%95%88)​
        
    - 구현이 용이하고 유연성이 높습니다.[swingswing.tistory](https://swingswing.tistory.com/43)​
        
    - 권한을 받은 사용자가 또 다른 사용자에게 권한을 넘겨줄 수 있어 보안에 취약할 수 있습니다.
        
- **대표적인 예**: SQL의 `GRANT`와 `REVOKE` 명령어가 DAC에 해당합니다.[swingswing.tistory](https://swingswing.tistory.com/43)​
    

## 2. 강제적 접근 통제 (MAC, Mandatory Access Control)

시스템 관리자가 미리 정해놓은 보안 정책(규칙)에 따라 접근 권한이 강제적으로 부여되는 방식입니다.[skby+1](https://blog.skby.net/db-%EC%A0%91%EA%B7%BC%EC%A0%9C%EC%96%B4-mac-dac-rbac/)​

- **통제 주체**: 시스템 (보안 관리자)
    
- **특징**:
    
    - 모든 주체(사용자)와 객체(데이터, 파일)에 **보안 등급(Security Level)**을 부여하고, 이 등급을 비교하여 접근을 제어합니다.[skby+1](https://blog.skby.net/db-%EC%A0%91%EA%B7%BC%EC%A0%9C%EC%96%B4-mac-dac-rbac/)​
        
    - 데이터 소유자라 할지라도 정책을 변경할 수 없으며, 시스템만이 접근을 통제합니다.[twilstudio](https://twilstudio.com/entry/%EC%A0%95%EB%B3%B4%EC%B2%98%EB%A6%AC%EA%B8%B0%EC%82%AC-%EB%8D%B0%EC%9D%B4%ED%84%B0%EB%B2%A0%EC%9D%B4%EC%8A%A4-%EB%B3%B4%EC%95%88)​
        
    - 보안성이 매우 높지만, 유연성이 떨어지고 관리 비용이 많이 듭니다.
        
- **대표적인 예**: 군사 기관이나 정부와 같이 높은 수준의 보안이 요구되는 환경에서 사용됩니다.[skby](https://blog.skby.net/db-%EC%A0%91%EA%B7%BC%EC%A0%9C%EC%96%B4-mac-dac-rbac/)​
    

## 3. 역할 기반 접근 통제 (RBAC, Role-Based Access Control)

사용자에게 직접 권한을 부여하는 대신, '역할(Role)'을 만들고 역할에 권한을 할당한 뒤 사용자를 해당 역할에 배정하는 방식입니다.[twilstudio+1](https://twilstudio.com/entry/%EC%A0%95%EB%B3%B4%EC%B2%98%EB%A6%AC%EA%B8%B0%EC%82%AC-%EB%8D%B0%EC%9D%B4%ED%84%B0%EB%B2%A0%EC%9D%B4%EC%8A%A4-%EB%B3%B4%EC%95%88)​

- **통제 주체**: 중앙 관리자
    
- **특징**:
    
    - 사용자의 직책이나 임무에 따라 역할을 부여하여 권한을 관리합니다.
        
    - 권한 관리가 단순해지고, 사용자의 보직 변경 시 역할만 변경하면 되므로 관리가 용이합니다.[swingswing.tistory](https://swingswing.tistory.com/43)​
        
    - DAC와 MAC의 단점을 보완한 방식으로, 현재 많은 시스템에서 채택하고 있습니다.[twilstudio](https://twilstudio.com/entry/%EC%A0%95%EB%B3%B4%EC%B2%98%EB%A6%AC%EA%B8%B0%EC%82%AC-%EB%8D%B0%EC%9D%B4%ED%84%B0%EB%B2%A0%EC%9D%B4%EC%8A%A4-%EB%B3%B4%EC%95%88)​
        
- **대표적인 예**: 회사에서 '회계팀', '인사팀', '개발팀' 등의 역할을 만들고 각 팀에 필요한 시스템 접근 권한을 부여하는 경우.

## 주요 기법 비교

|구분|임의적 접근 통제 (DAC)|강제적 접근 통제 (MAC)|역할 기반 접근 통제 (RBAC)|
|---|---|---|---|
|**통제 주체**|데이터 소유자|시스템|중앙 관리자|
|**정책 기반**|사용자(신원) 기반|보안 등급(규칙) 기반|역할(직책) 기반|
|**유연성**|높음|낮음|중간|
|**보안 수준**|낮음|높음|높음|
|**주요 사용처**|일반 상용 시스템 (SQL 등)|군사, 정부 기관|기업, 일반 조직|

## 주요 접근 통제 모델

특히 강제적 접근 통제(MAC)에는 다음과 같은 구체적인 모델이 있습니다.

- **벨-라파듈라 모델 (Bell-LaPadula)**: **기밀성**을 최우선으로 하는 모델입니다. 주체는 자신의 보안 등급보다 높은 등급의 객체를 읽을 수 없으며(No Read Up), 자신의 등급보다 낮은 등급의 객체에 쓸 수 없습니다(No Write Down).[swingswing.tistory](https://swingswing.tistory.com/43)​
    
- **비바 모델 (Biba)**: **무결성**을 최우선으로 하는 모델입니다. 주체는 자신의 무결성 등급보다 낮은 등급의 객체를 읽을 수 없으며(No Read Down), 자신의 등급보다 높은 등급의 객체에 쓸 수 없습니다(No Write Up).[swingswing.tistory](https://swingswing.tistory.com/43)​
