| 테스트 종류                         | 설명                                 | 수행 주체   | 특징                      |
| ------------------------------ | ---------------------------------- | ------- | ----------------------- |
| **단위 테스트 (Unit Test)**         | 프로그램의 최소 단위(함수, 모듈)가 올바르게 동작하는지 확인 | 개발자     | 화이트박스 성격 강함             |
| **통합 테스트 (Integration Test)**  | 모듈 간 인터페이스와 상호작용 검증                | 개발자/테스터 | 상향식, 하향식, 빅뱅 방식         |
| **시스템 테스트 (System Test)**      | 전체 시스템이 요구사항에 맞게 동작하는지 확인          | 독립된 테스터 | 실제 운영 환경과 유사하게 수행       |
| **인수 테스트 (Acceptance Test)**   | 사용자의 요구사항 충족 여부 확인                 | 고객/사용자  | 최종 승인 단계, 프로젝트 종료 판단 근거 |
| **알파 테스트 (Alpha Test)**        | 개발사 내부 환경에서 제한된 사용자 또는 QA팀이 수행     | 개발사 내부  | 베타 테스트 이전 단계            |
| **베타 테스트 (Beta Test)**         | 실제 사용자 환경에서 일반 사용자들이 참여            | 외부 사용자  | 출시 전 실제 사용 환경 검증        |
| **회귀 테스트 (Regression Test)**   | 수정·보완 후 기존 기능이 정상 동작하는지 확인         | 개발자/QA  | 자동화 테스트로 자주 수행          |
| **블랙박스 테스트 (Black Box Test)**  | 내부 구조는 보지 않고 입력과 출력만 검증            | 테스터     | 동치 분할, 경계값 분석 기법 활용     |
| **화이트박스 테스트 (White Box Test)** | 내부 로직과 코드 구조를 기반으로 검증              | 개발자     | 제어 흐름, 조건 커버리지 검사       |

##  핵심 요약

- **단위 → 통합 → 시스템 → 인수** 순으로 범위가 넓어짐
    
- **알파/베타 테스트**는 사용자 관점에서 제품 출시 전 검증
    
- **회귀 테스트**는 변경 후 기존 기능 영향 확인
    
- **블랙박스 vs 화이트박스**는 관점 차이
