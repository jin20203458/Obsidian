									
## 목적 기반 테스트 요약 및 보충 설명

|테스트 종류|설명|보충 및 수정|
|---|---|---|
|**회복 테스트** (Recovery Testing)|시스템에 의도적으로 실패를 유도하여, 정상적인 상태로 복귀하는지를 테스트합니다[rainbow96bear.tistory](https://rainbow96bear.tistory.com/entry/%EC%A0%95%EC%B2%98%EA%B8%B0-SW-%EA%B0%9C%EB%B0%9C-%EC%95%A0%ED%94%8C%EB%A6%AC%EC%BC%80%EC%9D%B4%EC%85%98-%ED%85%8C%EC%8A%A4%ED%8A%B8%EC%99%80-%EB%B6%84%EB%A5%98)​.|설명이 정확합니다. 백업/복구 절차, 재시작 시간 등을 검증합니다.|
|**안전 테스트** (Security Testing)|소스 코드 내의 보안 결함이나 외부의 불법적인 침입으로부터 시스템을 보호할 수 있는지를 테스트합니다[rainbow96bear.tistory](https://rainbow96bear.tistory.com/entry/%EC%A0%95%EC%B2%98%EA%B8%B0-SW-%EA%B0%9C%EB%B0%9C-%EC%95%A0%ED%94%8C%EB%A6%AC%EC%BC%80%EC%9D%B4%EC%85%98-%ED%85%8C%EC%8A%A4%ED%8A%B8%EC%99%80-%EB%B6%84%EB%A5%98)​.|설명이 정확합니다. SQL 인젝션, 크로스 사이트 스크립팅(XSS) 등의 취약점을 점검합니다.|
|**강도 테스트** (Stress Testing)|시스템에 과도한 부하(임계값 이상의 데이터나 트랜잭션)를 가하여 비정상적인 상황에서 어떻게 반응하는지를 테스트합니다[jbground.tistory+1](https://jbground.tistory.com/11)​.|설명이 정확합니다. 시스템의 한계점을 파악하고 장애 발생 시 동작을 확인하는 데 목적이 있습니다.|
|**성능 테스트** (Performance Testing)|사용자가 요구하는 응답 시간, 처리량, 자원 사용률 등을 만족하는지 테스트합니다[jbground.tistory+1](https://jbground.tistory.com/11)​.|설명이 정확합니다. 시스템이 실제 운영 환경에서 목표 성능을 보이는지 확인합니다.|
|**구조 테스트** (Structure Testing)|시스템 내부의 논리적 경로, 소스 코드의 복잡도 등을 평가하는 화이트박스 테스트 기법입니다[sungje365.tistory+1](https://sungje365.tistory.com/10)​.|설명이 정확합니다. 코드 커버리지(구문, 결정, 조건 등)를 측정하여 테스트의 충분성을 평가합니다.|
|**회귀 테스트** (Regression Testing)|시스템 변경(오류 수정, 기능 추가 등) 후, 이로 인해 기존 기능에 새로운 결함이 생기지 않았는지 확인하는 반복 테스트입니다.|설명이 정확합니다. 변경으로 인한 의도치 않은 부작용(Side effect)을 찾는 것이 주목적입니다.|
|**병행 테스트** (Parallel Testing)|`[사용자 설명]` 변경된 코드에 기존과 동일한 테스트 진행 후 결과 비교|**[수정된 설명]** 변경된 시스템과 기존 시스템에 동일한 데이터를 입력하여 **결과를 비교하고 일치하는지 검증**하는 테스트입니다.|

**병행 테스트(Parallel Testing)**는 단순히 '테스트를 진행 후 결과를 비교'하는 것 이상의 의미를 가집니다. 핵심은 **신규 시스템(또는 변경된 시스템)이 기존 시스템을 대체할 수 있을 만큼 동일하고 정확한 결과를 내는지 확인**하는 것입니다. 주로 시스템 마이그레이션이나 대규모 업데이트 시 신뢰성을 확보하기 위해 사용됩니다.

정리하자면, 제시된 내용은 대부분 정확하지만 '병행 테스트'는 '신/구 시스템 간의 결과 비교'라는 점을 명확히 하는 것이 중요합니다.
