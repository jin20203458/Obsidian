# SOLID 원칙 요약

**SOLID 원칙**은 객체지향 소프트웨어 설계에서 코드의 유지보수성과 확장성을 높이기 위해 지켜야 할 5가지 핵심 원칙입니다. 각 원칙의 앞글자를 따서 SOLID라고 부릅니다.[inpa.tistory+2](https://inpa.tistory.com/entry/OOP-%F0%9F%92%A0-%EA%B0%9D%EC%B2%B4-%EC%A7%80%ED%96%A5-%EC%84%A4%EA%B3%84%EC%9D%98-5%EA%B0%80%EC%A7%80-%EC%9B%90%EC%B9%99-SOLID)​

---

## 1. 단일 책임 원칙 (SRP: Single Responsibility Principle)

- **정의:** 한 클래스(모듈, 함수)는 오직 하나의 책임(변경 이유)만 가져야 한다.
    
- **핵심:** 한 가지 변경 사유로만 수정되어야 하므로, 여러 책임이 섞이지 않도록 분리해야 함.
    

## 2. 개방-폐쇄 원칙 (OCP: Open/Closed Principle)

- **정의:** 소프트웨어 요소(클래스, 모듈 등)는 **확장에는 열려 있고(변경 없이 새로운 기능 추가 가능), 수정에는 닫혀 있어야 한다(기존 코드 변경 최소화)**.
    
- **핵심:** 수정 대신 확장으로 새 기능을 추가하도록 설계.[alci+1](https://www.alci.dev/ko/que-es/solid)​
    

## 3. 리스코프 치환 원칙 (LSP: Liskov Substitution Principle)

- **정의:** **상위 타입(부모) 객체를 하위 타입(자식) 객체로 교체해도 프로그램이 정상 동작해야 한다.**
    
- **핵심:** 자식 클래스는 부모 클래스의 규칙과 동작을 반드시 지켜야 함.
    

## 4. 인터페이스 분리 원칙 (ISP: Interface Segregation Principle)

- **정의:** **필요하지 않은 기능까지 포함된 "거대한 인터페이스"보단, 여러 개의 작은 인터페이스로 나누어서** 클라이언트가 자신에게 필요한 기능만 사용하게 한다.
    
- **핵심:** 사용하지 않는 기능에 의존성 강요 금지, 인터페이스의 최소화.
    

## 5. 의존 역전 원칙 (DIP: Dependency Inversion Principle)

- **정의:** **상위 모듈(정책 결정)과 하위 모듈(구현 세부사항)이 서로 구체적인 것 대신 "추상(인터페이스/추상 클래스)"에 의존하도록 설계해야 한다.**
    
- **핵심:** 구체 클래스가 아니라, 추상 인터페이스 중심으로 의존 관계를 만든다.

## 요약 암기법

- **S**RP, **O**CP, **L**SP, **I**SP, **D**IP → SOLID
    
- 한글로: `단-개-리-인-의` (단일 책임, 개방-폐쇄, 리스코프 치환, 인터페이스 분리, 의존 역전)
