
## C# 프로그래밍 및 MVVM 패턴 학습 노트

## 1. C#의 속성(Property) 모델: C++과 다른 핵심적인 차이

C++에서는 `public` 멤버 변수에 직접 접근하는 것(`obj.x = 10;`)과 `public` 멤버 함수를 호출하는 것(`obj.setX(10);`)이 문법적으로 명확히 구분된다.

하지만 C#에서는 **속성(Property)** 이라는 특별한 언어 기능을 통해, 필드처럼 보이는 단순한 문법(`obj.X = 10;`)으로 실제로는 `set` 접근자(메서드)를 호출하는 것이 가능하다.

- **장점**:
    
    - **사용의 일관성**: 클래스 사용자는 내부 구현이 `public` 필드인지 복잡한 로직을 가진 속성인지 신경 쓸 필요 없이 동일한 문법을 사용한다.
        
    - **구현의 유연성**: 클래스 설계자는 나중에 `set` 접근자에 유효성 검사나 값 변경 알림 같은 로직을 추가하더라도, 외부 코드를 전혀 수정할 필요가 없다. 이는 C++에서 `public` 멤버를 `private`으로 바꿀 때 모든 관련 코드를 수정해야 하는 '리팩토링 지옥'을 방지한다.
        
- **자동 구현 속성 (Auto-Implemented Property)**:
    
    - `public string Name { get; set; }` 와 같이 한 줄로 선언하면, 컴파일러가 자동으로 보이지 않는 `private` 백킹 필드와 기본적인 `get`/`set` 로직을 생성해준다.
        
    - 이를 통해 개발자는 반복적인 상용구 코드 작성에서 해방되어 생산성이 극대화된다.
        

## 2. 소스 제너레이터(Source Generator)와 `partial` 키워드

- **`partial` 키워드**: 하나의 클래스 정의를 여러 파일에 나누어 작성할 수 있게 해주는 기능.
    
- **소스 제너레이터**: `[ObservableProperty]` 같은 어트리뷰트(Attribute)를 인식하여, 컴파일 시점에 필요한 C# 코드를 **동적으로 생성**하는 컴파일러 플러그인.
    
- **동작 원리**:
    
    1. 개발자가 `partial class` 안에 `[ObservableProperty] private string myVar;`를 선언한다.
        
    2. 소스 제너레이터가 이 어트리뷰트를 보고, `public string MyVar { get; set; }` 속성과 `PropertyChanged` 알림 로직이 포함된 새로운 `partial class` 파일을 생성한다.
        
    3. 컴파일러는 개발자가 작성한 원본 파일과 제너레이터가 생성한 파일을 **하나로 합쳐서** 최종 클래스를 컴파일한다.
        
    
    - 이 메커니즘 덕분에 `[ObservableProperty]` 같은 강력한 코드 자동화가 가능하다. `private` 필드는 생성된 `public` 속성의 **백킹 필드** 역할을 하므로 삭제되지 않고 그대로 유지된다.
        

## 3. MVVM(Model-View-ViewModel) 패턴의 핵심: 관심사의 분리

"구조가 똑같은데 왜 `CheckerItem`과 `CheckerSetting` 클래스를 따로 만들었을까?" 라는 질문에 대한 답.

- 이것은 MVVM 아키텍처의 핵심 철학인 **관심사의 분리(Separation of Concerns)**를 따르기 위한, 고도로 의도된 설계이다.
    

|구분|`CheckerSetting` (Model)|`CheckerItem` (ViewModel)|
|---|---|---|
|**목적**|**데이터 저장/로드**를 위한 순수 데이터 그릇|**UI 표시 및 상호작용**을 위한 기능적 객체|
|**책임**|데이터 구조 정의, 영구 저장소와 통신|UI 상태 관리, 사용자 입력 처리|
|**필요 기능**|단순한 속성(get/set)|**`ObservableObject` 상속** (값 변경 알림)  <br>**`[RelayCommand]`** (버튼 클릭 등 이벤트 처리)|
|**특징**|가볍고, 순수하며, UI 프레임워크에 독립적|UI와 밀접하게 연관된 기능들을 포함|

- **결론**: Model과 ViewModel을 분리함으로써, 데이터의 영구 저장 형식과 UI의 표현/동작 방식을 독립적으로 변경하고 발전시킬 수 있다. 이는 코드의 유연성, 테스트 용이성, 유지보수성을 극대화한다.
