MVVM(Model-View-ViewModel)은 프로그램을 세 부분으로 나눠서 개발하는 디자인 패턴, 즉 '역할 분담 규칙'입니다. 특히 WPF나 안드로이드처럼 화면 디자인과 프로그램 동작이 분리된 환경에서 코드를 깔끔하고 효율적으로 관리하기 위해 만들어졌습니다.

---

###  MVVM을 왜 쓸까? 

MVVM 없이 프로그램을 만들면 모든 코드가 한곳에 뒤섞이기 쉽습니다. 화면의 버튼 색깔 하나 바꾸려다가 데이터 처리 로직을 건드리는 실수를 할 수도 있죠.

MVVM은 **모델(Model), 뷰(View), 뷰모델**(ViewModel)이라는 세 명의 전문가에게 역할을 확실히 나눠줘서 이런 문제를 해결합니다.

---

## 세 명의 전문가: 
### Model : 보여줄 데이터의 원본
### View : 어떻게 보여줄 것인가?
### ViewModel : 무엇을 보여줄 것인가?

레스토랑에 비유하면 MVVM을 아주 쉽게 이해할 수 있습니다.

#### **① 뷰 (View): 손님에게 보이는 '메뉴판'과 '테이블'**

- **역할:** 사용자에게 **보여지는 모든 것**(UI)을 담당합니다. 버튼, 텍스트 상자, 이미지 등 화면의 모양과 디자인에만 집중합니다. WPF에서는 **XAML 코드**가 이 역할을 합니다.
    
- **레스토랑 비유:** 손님이 보는 예쁜 메뉴판, 테이블 세팅, 레스토랑 인테리어와 같습니다. 겉모습을 책임지죠.
    
- **규칙:** **절대로 스스로 데이터를 만들거나 계산하지 않습니다.** 그냥 뷰모델이 시키는 대로 보여주고, 손님(사용자)이 메뉴를 고르면(버튼 클릭) 그 사실을 뷰모델에게 알리기만 합니다.
    

#### **② 모델 (Model): 신선한 '재료'와 '레시피'**

- **역할:** 프로그램이 다루는 **실제 데이터와 핵심 로직**을 담고 있습니다. 예를 들어 '사용자' 정보(이름, 이메일)나 '상품' 정보(가격, 재고) 같은 순수한 데이터 덩어리와 그 데이터를 다루는 규칙(레시피)입니다.
    
- **레스토랑 비유:** 주방 냉장고에 있는 **신선한 재료(데이터)**나 이 재료로 요리하는 방법이 적힌 **기본 레시피**와 같습니다.
    
- **규칙:** **화면(View)이 어떻게 생겼는지 전혀 모릅니다.** 자신의 데이터가 어떻게 표시될지 전혀 신경 쓰지 않고, 오직 데이터 자체를 관리하는 데만 집중합니다.
    

#### **③ 뷰모델 (ViewModel): 유능한 '지배인'**

- **역할:** **View와 Model 사이의 중재자** 역할을 하는 가장 중요한 부분입니다. View가 보여줘야 할 데이터를 Model로부터 가져와서 보기 좋게 가공하고, View에서 사용자가 무언가를 클릭하면(예: "주문하기" 버튼), 이를 해석해서 Model의 데이터를 변경하도록 지시합니다.
    
- **레스토랑 비유:** 손님(View)과 주방(Model) 사이를 오가는 **지배인**입니다. 손님이 메뉴판을 보고 "스테이크 주세요"라고 하면(View의 요청), 지배인은 주방에 "스테이크 하나 준비해줘"라고 **주문(명령)**을 넣습니다(Model에 전달). 또, 주방에서 요리가 다 되면(Model의 데이터 변경), 지배인이 이걸 예쁘게 접시에 담아(데이터 가공) 손님 테이블에 가져다줍니다(View에 전달).
    
- **규칙:** **View가 누구인지(버튼인지, 텍스트인지) 구체적으로 알지 못합니다.** 그저 "주문하기 기능", "메뉴 보여주기 기능" 같은 **명령(Command)과 데이터**만 준비해 놓습니다. View는 이 기능과 데이터를 가져다 쓸 뿐입니다.
    

---

### ## MVVM의 핵심 기술: 데이터 바인딩과 커맨드

뷰모델(지배인)이 뷰(손님)와 직접 대화하지 않고도 일을 처리하는 비결은 **데이터 바인딩**과 **커맨드**라는 마법 같은 기술 덕분입니다.

- **데이터 바인딩 (Data Binding):** 뷰모델의 데이터와 뷰의 UI 요소를 '보이지 않는 끈'으로 연결하는 것입니다. 뷰모델의 '오늘의 메뉴' 데이터가 "피자"에서 "파스타"로 바뀌면, 뷰의 메뉴판 글씨도 **자동으로** "파스타"로 바뀝니다. 지배인이 일일이 메뉴판을 고쳐 쓸 필요가 없죠.
    
- **커맨드 (Command):** 뷰에서 일어나는 행동(예: 버튼 클릭)을 뷰모델의 특정 기능에 연결하는 것입니다. 사용자가 '주문하기' 버튼을 누르면, 미리 연결된 뷰모델의 '주문 접수' 기능이 **자동으로** 실행됩니다.
    

### ## MVVM을 사용하면 좋은 점 ✨

1. **협업이 쉬워져요:** 디자이너는 View(XAML)만, 개발자는 ViewModel(C#)과 Model(C#)에 집중할 수 있습니다.
    
2. **코드가 깔끔해져요:** 각자의 역할이 명확해서 코드를 이해하고 수정하기(유지보수)가 매우 쉬워집니다.
    
3. **테스트가 쉬워져요:** UI 없이도 ViewModel의 로직이 잘 동작하는지 독립적으로 테스트할 수 있습니다.
    
4. **코드를 재사용하기 좋아요:** 하나의 ViewModel을 만들어두면, 약간 다른 디자인의 View에도 쉽게 재사용할 수 있습니다.
