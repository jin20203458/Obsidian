Clang Static Analyzer (CSA) 체커 개발은 일반적인 C++ 프로그램 디버깅과는 다른 접근 방식이 필요합니다. CSA는 심볼릭 실행(Symbolic Execution)을 기반으로 코드의 여러 경로를 탐색하므로, 디버깅의 핵심은 **"분석기가 현재 어떤 경로를 탐색 중인지", "특정 변수의 상태(State)가 무엇인지", "AST(추상 구문 트리)가 어떻게 구성되어 있는지"** 등을 파악하는 것입니다.

---
## 1. 분석기 출력 및 시각화 (핵심)

가장 중요하고 CSA에 특화된 디버깅 방법입니다. 분석기가 탐색하는 경로(Exploded Graph)를 직접 확인하는 것입니다.

### `debug.ViewExplodedGraph` 체커 사용

- **기능:** 분석기가 생성하는 '폭발 그래프(Exploded Graph)'를 실시간으로 시각화해 줍니다. 각 노드는 프로그램의 상태를, 엣지(간선)는 제어 흐름(분기, 함수 호출 등)을 나타냅니다.
    
- **사용법:** 분석을 실행할 때 `debug.ViewExplodedGraph` 체커를 활성화합니다. (Graphviz `dot` 프로그램이 설치되어 있어야 합니다.)
    
    Bash
    
    ```
    # 특정 체커(my.checker)와 함께 폭발 그래프를 봅니다.
    clang -cc1 -analyze -analyzer-checker=my.checker,debug.ViewExplodedGraph test.c
    ```
    
- **장점:**
    
    - 내 체커가 **어떤 경로를 탐색하고, 어떤 경로를 가지치기(pruning)했는지** 시각적으로 바로 알 수 있습니다.
        
    - 경로 상에서 **상태(State)가 어떻게 변하는지** (예: `x = 5` -> `x = 6`) 추적할 수 있습니다.
        
    - 버그 리포트가 생성되는 지점(Sink Node)을 명확히 볼 수 있습니다.
        

### `debug.DumpExplodedGraph` 체커 사용

- **기능:** `ViewExplodedGraph`와 동일하지만, 그래프를 시각화하는 대신 콘솔에 텍스트(DOT 형식)로 덤프합니다.
    
- **사용법:**
    
    Bash
    
    ```
    clang -cc1 -analyze -analyzer-checker=my.checker,debug.DumpExplodedGraph test.c
    ```
    
- **장점:** GUI 환경이 아니거나, 텍스트 로그로 분석 결과를 저장하고 싶을 때 유용합니다.

## 2. AST 및 CFG 덤프

체커가 탐색을 시작하기 _전_에, Clang이 코드를 어떻게 해석했는지 확인하는 단계입니다. 체커가 특정 `Stmt`나 `Decl`을 방문(visit)하지 않는다면, 대부분 AST 구조가 예상과 다르기 때문입니다.

### AST (Abstract Syntax Tree) 덤프

- **기능:** 소스 코드가 파싱된 후의 트리 구조를 덤프합니다.
    
- **사용법:**
    
    Bash
    
    ```
    clang -Xclang -ast-dump -fsyntax-only test.c
    ```
    
- **장점:**
    
    - 내가 찾으려는 `CallExpr` (함수 호출), `BinaryOperator` (이항 연산) 등이 AST 상에서 어떤 부모-자식 관계를 갖는지 정확히 알 수 있습니다.
        
    - `ImplicitCastExpr` (암시적 형변환)처럼 코드에 드러나지 않는 노드들을 확인할 수 있어 매칭(matching) 시 실수를 줄여줍니다.
        

### CFG (Control Flow Graph) 덤프 및 시각화

- **기능:** 함수의 제어 흐름 그래프(기본 블록과 그 사이의 관계)를 덤프하거나 시각화합니다.
    
- **사용법:**
    
    Bash
    
    ```
    # CFG를 콘솔에 덤프
    clang -cc1 -analyze -analyzer-checker=debug.DumpCFG test.c
    
    # CFG를 시각화 (Graphviz 필요)
    clang -cc1 -analyze -Xclang -view-cfg test.c
    ```
    
- **장점:** `checkBranchCondition` 등 분기 관련 로직을 디버깅할 때, CFG가 어떻게 구성되었는지 확인하면 큰 도움이 됩니다.

## 3. 코드 내 디버깅 (printf 스타일)

체커 코드 내에 직접 디버깅 구문을 삽입하여, 특정 지점을 지날 때의 상태나 값(Value)을 출력합니다.

### `llvm::dbgs()` 사용

- **기능:** LLVM/Clang에서 사용하는 표준 디버그 출력 스트림입니다. (`stderr`로 출력됩니다.)
    
- **주의:** **`std::cout`이나 `printf`를 사용하면 안 됩니다.** Clang은 `stdout`을 다른 용도(예: `preprocessed`된 코드 출력)로 사용하며, 버퍼링 문제로 인해 출력이 꼬일 수 있습니다.
    
- **사용법:**
    
    C++
    
    ```
    #include "llvm/Support/Debug.h" // 또는 llvm/Support/raw_ostream.h
    
    void MyChecker::checkPreStmt(const Stmt *S, CheckerContext &C) const {
        // ...
        llvm::dbgs() << "Visiting Stmt: " << S->getStmtClassName() << "\n";
    }
    ```
    

### `dump()` 메소드 활용

- **기능:** Clang/LLVM의 거의 모든 주요 객체( `Stmt`, `Decl`, `QualType`, `SVal`, `ProgramStateRef` 등)는 자신을 덤프하는 `.dump()` 메소드를 제공합니다.
    
- **사용법:**
    
    C++
    
    ```
    void MyChecker::checkPreStmt(const Stmt *S, CheckerContext &C) const {
        // 현재 Stmt의 AST를 덤프
        S->dump(); 
    
        // 현재 분석 상태(State)를 덤프
        ProgramStateRef State = C.getState();
        State->dump();
    
        // 특정 심볼릭 값(SVal)을 덤프
        SVal V = State->getSVal(S, C.getLocationContext());
        V.dump();
    }
    ```
    
- **장점:** `llvm::dbgs()`로 수동 출력하는 것보다 훨씬 자세하고 구조화된 정보를 제공합니다.

## 4. GDB/LLDB를 이용한 스텝 디버깅

체커 로직이 복잡해져서 상태 변화를 정밀하게 추적해야 하거나, 체커 자체가 크래시(Crash)가 날 때 사용합니다.

- **핵심:** `scan-build`나 `clang` 명령어를 직접 디버깅하는 것은 비효율적입니다. Clang 드라이버는 실제 분석을 수행하는 `clang -cc1`이라는 자식 프로세스를 실행하기 때문입니다.
    
- **디버깅 절차:**
    
    1. 정확한 clang -cc1 명령어 확보:
        
        scan-build나 clang 명령어에 -v (verbose) 옵션을 추가하여 실행하면, 실제 내부적으로 실행되는 clang -cc1 ... 명령어가 전부 출력됩니다.
        
        Bash
        
        ```
        scan-build -v make # 또는 clang -v -analyze ... test.c
        ```
        
    2. clang -cc1 명령어 복사:
        
        출력된 로그에서 clang -cc1 -analyze ... (매우 긴 인수들) ... test.c 부분을 통째로 복사합니다.
        
    3. **GDB/LLDB 실행:**
        
        - **주의:** Clang/LLVM을 **디버그 빌드(Debug Build)**로 컴파일해야 제대로 디버깅할 수 있습니다. (Release 빌드는 최적화 때문에 변수 확인 등이 어렵습니다.)
            
        
        Bash
        
        ```
        # GDB
        gdb --args [복사한 clang -cc1 명령어 전부 붙여넣기]
        
        # LLDB
        lldb -- [복사한 clang -cc1 명령어 전부 붙여넣기]
        ```
        
    4. **브레이크포인트 설정 및 실행:**
        
        코드 스니펫
        
        ```
        (gdb) b MyChecker::checkPreStmt # 중단점 설정
        (gdb) run
        ```

## 5. 유용한 체커 설정 및 팁

### 특정 체커만 실행하기

디버깅 시에는 다른 체커들이 불필요한 노이즈를 만들 수 있습니다. 내 체커만 실행하는 것이 좋습니다.

Bash

```
clang -cc1 -analyze \
    -analyzer-disable-all-checks \  # 모든 기본 체커 비활성화
    -analyzer-checker=my.checker,debug.ViewExplodedGraph \ # 내 체커와 디버그 체커만 활성화
    test.c
```

### `debug` 그룹의 다른 체커들

- `debug.DumpLiveVars`: 각 기본 블록(Basic Block)의 시작과 끝에서 살아있는 변수(Live Variables)를 덤프합니다.
    
- `debug.DumpCallGraph`: 호출 그래프(Call Graph)를 덤프합니다.
    

이 방법들을 조합하여 사용하면, 분석기가 코드를 어떻게 해석하고(AST/CFG), 어떤 경로를 탐색하며(Exploded Graph), 각 단계에서 상태가 어떻게 변하는지(dump, llvm::dbgs) 효과적으로 파악할 수 있습니다.
