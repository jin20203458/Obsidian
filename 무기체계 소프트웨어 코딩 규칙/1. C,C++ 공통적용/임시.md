좋아. 아래 항목들은 원래 **CSA(경로 민감 분석)**으로 생각하기 쉬운 주제지만, **린터(AST/CFG 기반)** 만으로도 “쓸 만한 품질”의 커버리지를 꽤 확보할 수 있어. 각 룰마다 **대안 설계(린터)**, 적용 가능한 **분석 깊이(Tier)**, **권장 구현**과 **CSA로 상향해야 할 경계**를 정리했어.

> 표기  
> **Tier‑A(AST)**: 순수 AST 규칙·상수/타입 기반 · 패턴 매칭  
> **Tier‑B(AST+CFG)**: 얕은 데이터플로(지배/도달·구간제약) · 함수 내 경로 상식 수준  
> **Tier‑C(CSA)**: 경로 민감(루프 누적, 별칭, 다중 경로 결합 등)

---

## 1) **값이 초기화되지 않은 변수의 포인터를 Read‑only(const)로 사용 금지**

**핵심**: `&var` 또는 `ptr`이 **초기화 전**에 `const T*` 인자로 전달되는 호출

- **Tier‑A(AST)**
    
    - 패턴: `callExpr(callee(functionDecl(hasParameter( hasType(pointerType(pointee(isConstQualified())))))) , hasArgument(N, unaryOperator(hasOperatorName("&"), hasUnaryOperand(declRefExpr(to(varDecl().bind("V"))))))`
        
    - 한계: _초기화 여부_를 모름 → 진단 보류용 캐치만 가능
        
- **Tier‑B(AST+CFG) — 권장** ✅
    
    - **함수 내부 단정 초기화(Definite‑assignment)**: `VarDecl V`의 선언 지점~호출 지점까지 **할당/초기화 이벤트**(=, InitList, `memset(&V,…)` 등)를 탐색. 없으면 경고.
        
    - 구현 포인트:
        
        - CFG 블록 순서 + **도달 가능 경로 모두에서 “할당 없음”**일 때만 보고(보수적).
            
        - 간단한 소거 규칙: `if` 분기 한쪽에서만 대입 → **불확정**으로 간주해 미보고.
            
    - 클린 업: Clang‑Tidy의 `utils`에 있는 간이 데이터플로/도미네이터 헬퍼를 재사용(또는 간단 Map으로 블록별 “정의됨/정의 안됨” 상태 전파).
        
- **Tier‑C(CSA)**
    
    - 루프 누적/별칭(`ptr = &V; …; f(ptr)`)까지 정확히 보려면 필요.
        
    - 초기 릴리스는 B로 충분. **오탐 회피가 중요**(불확정 경로는 미보고).
        

---

## 2) **수행되지 않는(도달 불가) 소스 코드 금지**

**핵심**: dead code — `return`/`break` 이후, 항상 거짓 분기, 불가능한 `case` 등

- **Tier‑A(AST)**
    
    - 우리는 이미 **항상 참/거짓 조건**과 **불가능한 switch case**를 커버하는 체커를 설계/구현했음.
        
- **Tier‑B(AST+CFG) — 권장** ✅
    
    - CFG에서 **도달 불가 블록**(unreachable) 또는 **terminator 이후 문장** 탐지.
        
    - `CFGBlock::succ_empty`, `Stmt 후단 terminator 확인`으로 간단히 가능.
        
- **CSA**: 불필요.
    

---

## 3) **표현 불가능한 데이터로의 형 변환 금지(값 손실 캐스트)**

**핵심**: **축소 변환(narrowing)**, **부호 변경**으로 표현 영역 이탈 가능

- **Tier‑A(AST) — 권장(1단계) ✅**
    
    - **상수식**은 확정 판정: `int x = (short)100000;` → 즉시 경고(APSInt 비교).
        
    - 타입 규칙 기반 의심 패턴:
        
        - 더 넓은 → 더 좁은 정수폭 (e.g., 64→32)
            
        - 부호 있는 → 부호 없는(또는 반대)로 **명시 캐스트**
            
    - 옵션으로 “상수만 엄격, 비상수는 ‘의심’ 레벨” 구분.
        
- **Tier‑B(AST+CFG)**
    
    - 변수값에 **상한/하한 제약**이 앞에서 주어졌다면(예: `0<=x && x<=255`) 그 안에서만 허용.
        
    - 간단한 **구간제약 스택**(우리가 Always‑Constant에서 쓴 방식) 재사용 가능.
        
- **CSA**
    
    - 다중 경로 누적·루프 축약이 필요하면 선택.
        

---

## 4) **포인터 캐스트로 인한 const 상실 주의**

**핵심**: `const T*` → `T*` 로 **qualifier drop**

- **Tier‑A(AST) — 권장 ✅**
    
    - `ExplicitCastExpr`(C‑style, `static_cast`, `reinterpret_cast`, `const_cast`)와 **암시 변환**에서 `pointee`의 `Qualifiers` 비교.
        
    - `SrcPointee.isConstQualified() && !DstPointee.isConstQualified()`면 진단.
        
    - 옵션: 허용 목록(라이브러리 내부, `memcpy` 래퍼 등)로 소음 제어.
        
- **CSA**: 불필요.
    

---

## 5) **포인터 캐스트로 인한 volatile 상실 주의**

- **Tier‑A(AST) — 권장 ✅**
    
    - const와 동일 로직으로 `isVolatileQualified()` 비교.
        
- **CSA**: 불필요.
    

---

## 6) **포인터는 참조 전에 NULL 확인**

**핵심**: `*p`, `p->` 사용 전에 **null‑check 도미네이트** 필요

- **Tier‑A(AST)**
    
    - 패턴만으로 확정하기 어려움.
        
- **Tier‑B(AST+CFG) — 권장 ✅**
    
    - **도미네이터 기반 근접 가드**: 동일 기본 블록 또는 직전 지배 블록에 `p != 0`/`if(p)`가 **모든 경로**에 존재하는지 확인.
        
    - 얕은 규칙: “**같은 블록** 안에서 `if (!p) return;` 후 사용” → OK.
        
    - 별칭·재할당(예: `p = q`) 후 사용은 보수적으로 미보고.
        
- **Tier‑C(CSA)**
    
    - 함수 포인터/별칭/루프 수정까지 정밀하게 보려면 필요(선택).
        

---

## 7) **배열 인덱스가 선언 크기 초과 금지**

- **Tier‑A(AST) — 권장(부분) ✅**
    
    - **상수 인덱스**는 확정 판정: `int a[10]; a[10]`/`a[-1]` → 경고.
        
    - `std::array<N>`의 템플릿 N도 상수로 취급.
        
- **Tier‑B(AST+CFG)**
    
    - **루프 패턴**: `for(i=0;i<N;i++) a[i]` 등 전형식은 안전으로 표시(소음 제거).
        
    - 간단 제약: 앞선 조건 `0 <= idx && idx < N`이 도미네이트하면 허용.
        
- **CSA**
    
    - 비전형 패턴, 다중 경로 결합, 인덱스 연산 누적이면 선택.
        

---

## 8) **Null Pointer 산술 금지**

- **Tier‑A(AST)**
    
    - 상수식: `((int*)0)+1`, `NULL+1`, `++((T*)0)` → 확정 경고.
        
- **Tier‑B(AST+CFG) — 권장 ✅**
    
    - **분기 내**: `if (p == NULL) { p++; }` 같은 **명백한 위반**을 탐지(현재 경로 제약에서 p==0이면 포인터 산술 사용 금지).
        
    - 얕은 제약이면 충분히 유용.
        
- **CSA**: 굳이 필요 없음.
    

---

## 9) **나눗셈에서 피제수가 변수면 0 확인 필수**

- **Tier‑A(AST)**
    
    - 상수 0은 컴파일러가 잡음.
        
- **Tier‑B(AST+CFG) — 권장 ✅**
    
    - `/ var` 또는 `% var` 직전 경로에 `var != 0` 또는 `var` truthy 체크가 **모든 경로**에서 도미네이트하는지 확인.
        
    - 흔한 패턴만 허용 처리(같은 블록 또는 직전 지배).
        
- **CSA**
    
    - 다중 경로 결합·루프 갱신까지 완전 정밀도를 원하면 선택.
        

---

## 10) **시프트 연산: 음수 or 데이터 폭 초과 금지**

- **Tier‑A(AST) — 권장(1단계) ✅**
    
    - `lhs << k`/`>> k`에서 **상수 k**가 `<0` 또는 `>= width(lhs‑type)` → 경고.
        
    - width 구하기: 정수승격 후 **실제 적용 폭** 사용(플랫폼 일관성).
        
- **Tier‑B(AST+CFG)**
    
    - 변수 `s`일 때, **상위 제약**으로 `0 <= s && s < width` 확인 시 허용.
        
    - 루프 내 `for(s=0; s<width; ++s)` 같은 전형 패턴 화이트리스트.
        
- **CSA**: 선택.
    

---

## 11) **서명 있는 값의 비트 연산으로 인한 Negative Value 유의**

- **현실적 정책**
    
    - MISRA류 스타일에 맞춰 **“비트 연산은 부호 없는 정수에만”** 규칙으로 단순화하는 게 **정확도/소음/가독** 모두에 유리.
        
- **Tier‑A(AST) — 권장 ✅**
    
    - `~ | ^ & << >>`의 피연산자가 **signed 정수**면 경고(옵션화).
        
    - 예외: `>>` 산술시프트를 명시적으로 허용할지 옵션으로 제어.
        
    - 더 섬세하게 하려면 “결과가 음수 컨텍스트로 소비되는가”까지 보지만 소음 ↑.

## 구현 팁(공통)

- **제약 스택 재사용**: 우리가 만든 `AlwaysConstantConditionCheck`의 **정수 구간 제약(Interval) 추출/적용기**를 그대로 재사용하면,  
    null‑check(`p != 0`), 0 나눗셈(`d != 0`), 인덱스 범위(`0 <= i && i < N`), 시프트 범위 등 **동일 틀**로 해결 가능.
    
- **지배/도달 조건만** 본다(성능·소음 균형). 합류점·복잡 루프는 “미결정=미보고”.
    
- **옵션으로 소음 제어**:
    
    - `OnlyMainFile`, `IgnoreSystemHeaders`, `IgnoreMacroExpansions`
        
    - `UseParentDominatorOnly`, `MaxConstraintDepth`, `RequireBothBoundsForRange` 등
        
- **명백 케이스부터**: 항상 참/거짓, 상수 위반, 타입 불변 위반 → 신뢰도 높은 규칙을 먼저 배치하고, 경로 제약은 보수적으로.
    

---

### 결론

말한 항목들 대부분은 **CSA 없이**도 **AST/CFG 기반 린터**로 **실용적 정확도**를 낼 수 있어.  
초기 버전은 **Tier‑A/B로 릴리스**하고, 프로젝트에서 정말 필요한 지점에서만 **CSA를 선택적으로** 켜는 구성이 **속도·정확도·유지보수성** 균형이 가장 좋다.

원하면, 위 11개 룰 중 우선순위가 높은 3~4개부터 **체커 스켈레톤(매처 + 핵심 판정 루틴)**까지 바로 써서 넘길게.
