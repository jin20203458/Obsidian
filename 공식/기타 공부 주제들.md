처음 보면 다소 생소하고 대단히 도전적이었을 핵심 기술 스택 5가지를 정리해 드립니다.

---

### 1. EnTT (데이터 지향 Entity Component System)
객체 지향(OOP)의 고정관념을 깨는 **데이터 지향 설계(Data-Oriented Design)** 라이브러리입니다.

*   **도전적인 이유**: 전통적인 C++ 방식인 `class NPC { Position pos; Emotion emo; void Move(); }` 같은 설계 대신, 모든 데이터를 쪼개어 배열 형태로 메모리에 연속적으로 배치합니다.
*   **어떤 이득이 있나요?**: CPU가 메모리에서 데이터를 읽어올 때 발생하는 **캐시 미스(Cache Miss)**를 최소화하여, NPC 수가 수백, 수천 명으로 늘어나도 미친 수준의 메모리 접근 속도와 연산 능력을 보장합니다.

---

### 2. Boost.Asio & C++20 코루틴 (`co_await`)
비동기 소켓 네트워크 프로그래밍의 정점이며, 최신 C++ 문법의 정수입니다.

*   **도전적인 이유**: 보통 비동기 서버는 콜백 함수가 꼬리에 꼬리를 무는 "콜백 지옥"이나 복잡한 멀티스레드 동기화(Lock) 늪에 빠지기 쉽습니다. 우리는 **C++20 표준 코루틴**인 `co_await`와 Boost.Asio의 `awaitable`을 결합해 비동기 코드를 작성했습니다.
*   **어떤 이득이 있나요?**: 네트워크가 백그라운드에서 비동기로 수신되지만, 실제 코드는 마치 위에서 아래로 순차 실행되는 동기식 코드처럼 읽힙니다. 성능(비동기)과 가독성(동기식 직관성)을 모두 챙긴 고급 기술입니다.

---

### 3. gRPC 비동기 래퍼 & CompletionQueue 마샬링
gRPC C++의 비동기 API는 난이도가 매우 높기로 악명이 높습니다.

*   **도전적인 이유**: gRPC의 비동기 호출은 `CompletionQueue`에 완료 신호가 들어왔을 때 단순히 `void*` 형식의 가짜 포인터(Tag)만 던져줍니다. 이 날것의 통신 체계를 객체 지향 콜백으로 변환하기 위해, C++ 템플릿 구조체(`RpcCall<TResp>`)와 다형성 구조(`AsyncRpcTag`)를 직접 커스텀 구현했습니다.
*   **어떤 이득이 있나요?**: 비동기 네트워크 처리가 끝나면 주 메모리 스레드로 안전하게 결과를 전달(마샬링)하는 락프리 디스패치 파이프라인을 구축해, 멀티스레드 충돌 없이 안전하게 C# 응답을 수용할 수 있게 되었습니다.

---

### 4. 공간 분할 격자판 (Spatial Hash Grid) 구현
대규모 멀티플레이 서버의 필수 수학/알고리즘 기술입니다.

*   **도전적인 이유**: 모든 유저와 NPC가 "내 주변에 누가 있지?"를 물어볼 때 전체 대상을 다 훑으면 연산량이 기하급수적(O(N²))으로 폭발합니다. 이를 해결하기 위해 2D 세상을 바둑판 형태의 격자로 나누는 알고리즘을 자체 구현했습니다.
*   **어떤 이득이 있나요?**: 주변 NPC를 찾는 연산 성능이 획기적으로 개선되었고, 특히 스텝 3에서 삽입/이동 연산을 보조 인덱스 해시맵을 사용해 **O(1)**로 극대화하여 엔진 스레드 성능을 완전히 지켜냈습니다.

---

### 5. 엄격한 C++20 표준 및 최신 MSVC (VS 2026 / v18) 대응
최신 컴파일러의 엄격한 형식 검사를 통과한 안정성 확보 작업입니다.

*   **도전적인 이유**: 최신 컴파일러 환경에서는 데이터 손실 가능성이 있는 암시적 형변환(예: `int`를 `uint32_t`로 변환)에 대해 컴파일 오류(C2397 등)를 뱉으며 빌드를 차단합니다.
*   **어떤 이득이 있나요?**: 난수 범위 지정이나 타입 변환 시 `0u` 리터럴을 명시하고 정밀하게 캐스팅 코드를 수정함으로써, 런타임에 발생할 수 있는 뜬금없는 오버플로우나 크래시 버그를 빌드 단계에서 원천 봉쇄했습니다.
