# Phase 5-2.D+: 소문 기억 공식 리팩토링 (공식 C: 심리 몰입형)

현재 Top-K 소문 압축 공식이 **당사자 가중치(+0.5)에 과도하게 쏠려** 기억 슬롯 3개를 당사자 소문이 독점하는 구조적 문제가 발견되었습니다.
또한 **폭로할 소문(`gossipToShare`)이 Top-K 기억 목록 밖에서 선택**되어, NPC가 자기 머릿속에 없는 소문을 지시받아 말하게 되는 **인지 부조화(Cognitive Dissonance)** 버그가 존재합니다.

본 계획서는 두 가지 문제를 동시에 해결합니다:
1. **공식 C (심리 몰입형)** 점수 공식으로 교체 — 관계 감정과 와전 자극도를 반영한 NPC 개성 분화
2. **대안 B (기억 내 전파 선택)** 흐름으로 리팩토링 — Top-K 기억 안에서 전파할 소문을 고르는 인지적 일관성 확보

> - NPC가 머릿속에 없는 소문을 지시받는 상황이 완전히 제거됩니다.

## 공식 C: 심리 몰입형 점수 공식 (최종 권장치)

```
Score = SubjectiveBelief                                    // 기본 확신도 (0.0 ~ 1.0)
      + (Subject == Listener ? 0.15 : 0)                    // 당사자 인식 가중치
      + (!HasSharedWithOthers ? 0.25 : 0)                    // 미전파 신선 소문 가중치
      + min(abs(Liking_toward_subject) / 250.0, 0.4)         // 감정 연동 가중치
      + min(MutationCount × 0.08, 0.25)                      // 와전 자극도 가중치
      + max(0, (1.0 - secondsSinceAcquired / 1000.0) × 0.2)  // 시간 신선도 가중치
```

### 각 항목 상세 설명

| 항목 | 범위 | 설명 |
|---|---|---|
| **SubjectiveBelief** | `0.0 ~ 1.0` | 소문을 얼마나 확실히 믿는가. 기존과 동일 |
| **당사자 인식** | `0` 또는 `+0.15` | 대화 상대방 본인에 관한 소문은 살짝 의식. 기존(+0.5)의 30%로 대폭 하향 |
| **미전파 가중치** | `0` 또는 `+0.25` | 아직 남에게 한 번도 안 말한 따끈한 소문. 기존(+0.1)의 2.5배 상향. 전파 병목 방지 핵심 |
| **감정 연동** | `0.0 ~ 0.4` | 소문 대상에 대한 NPC의 감정이 강할수록(사랑이든 미움이든) 해당 소문이 머릿속에 강하게 남음. `abs(Liking)/250` 계산, 상한 0.4 |
| **와전 자극도** | `0.0 ~ 0.25` | 여러 입을 거쳐 과장/왜곡된 소문일수록 자극적이어서 기억에 남음. `MutationCount × 0.08`, 상한 0.25 |
| **시간 신선도** | `0.0 ~ 0.2` | 최근에 들은 소문이 기억에 더 잘 떠오름. 기존과 동일 |

### 시뮬레이션 예시: 에바(Liking: 카일=-40, 바르트=+10)가 카일과 대화

| 소문 | Belief | 당사자 | 미전파 | 감정 | 와전 | 신선도 | **총점** |
|---|---|---|---|---|---|---|---|
| 카일 성수 절도 | 0.90 | +0.15 | — | +0.16 | +0.08 | +0.10 | **1.39** |
| 바르트 마법석 | 0.70 | — | +0.25 | +0.04 | — | +0.20 | **1.19** |
| 촌장 뇌물 | 0.85 | — | — | +0.12 | +0.16 | +0.05 | **1.18** |
| 케빈 외상 | 0.50 | — | +0.25 | — | +0.24 | — | **0.99** |

**결과 (Top-3 기억):**
1. 🔴 카일 성수 절도 (1.39) — 속으로 의심하며 긴장
2. 🟢 바르트 마법석 (1.19) — **전파 후보로 당첨** (당사자가 아닌 미전파 소문)
3. 🟡 촌장 뇌물 (1.18) — 배경 인지

## 대안 B: 기억 내 전파 선택 흐름

### 기존 흐름 (인지 부조화 발생 가능)

```
1. SelectGossipToShare() → 전체 소문에서 전파할 소문 1개 선택
2. Top-K 압축 → 기억 슬롯 3개 독립 선정
3. 시스템 프롬프트 조립 → ❌ 1번 결과가 2번 결과에 없을 수 있음
```

### 신규 흐름 (인지 일관성 보장)

```
1. Top-K 압축 (공식 C) → 기억 슬롯 3개 확정
2. 기억 3개 중에서 전파할 소문 선택 (당사자 제외 + 외향성 확률 체크)
3. 시스템 프롬프트 조립 → ✅ 전파 지시 소문이 반드시 기억 안에 있음
```

### DialogueOrchestrator.cs

#### [MODIFY] [DialogueOrchestrator.cs](file:///C:/Users/adg01/Documents/GitHub/MundusVivens/MundusVivens.Prototype/Services/DialogueOrchestrator.cs)

**변경 1: Top-K 점수 공식을 공식 C로 교체**

기존 점수 계산 블록(약 L103~L123)을 공식 C 연산으로 완전 교체합니다.
감정 가중치 계산을 위해 `currentSpeaker.RelationshipMap`에서 소문 대상에 대한 `Liking` 값을 조회합니다.

```csharp
var sortedGossips = currentSpeaker.KnownGossips.Values
    .Select(kg => {
        double score = kg.SubjectiveBelief;

        // 당사자 인식 가중치
        if (kg.Gossip.Subject == currentListener.AgentId)
            score += 0.15;

        // 미전파 소문 가중치
        if (!kg.HasSharedWithOthers)
            score += 0.25;

        // 감정 연동 가중치
        if (currentSpeaker.RelationshipMap.TryGetValue(kg.Gossip.Subject, out var subjectRel))
            score += Math.Min(Math.Abs(subjectRel.Liking) / 250.0, 0.4);

        // 와전 자극도 가중치
        score += Math.Min(kg.Gossip.MutationCount * 0.08, 0.25);

        // 시간 신선도 가중치
        double secondsSinceAcquired = (DateTime.UtcNow - kg.AcquiredAt).TotalSeconds;
        score += Math.Max(0.0, (1.0 - secondsSinceAcquired / 1000.0) * 0.2);

        return new { Gossip = kg, Score = score };
    })
    .OrderByDescending(x => x.Score)
    .Take(3)
    .Select(x => x.Gossip)
    .ToList();
```

**변경 2: 전파 소문 선택을 Top-K 결과 기반으로 전환**

기존 흐름:
```csharp
// 대화 루프 바깥에서 전체 소문으로부터 선택
var gossipToShareByA = _gossipEngine.SelectGossipToShare(agentA, agentB);
```

신규 흐름:
```csharp
// 1단계: Top-K 기억 확정 (대화 루프 바깥으로 이동)
var topKForA = ComputeTopK(agentA, agentB);  // 공식 C 적용
var topKForB = ComputeTopK(agentB, agentA);  // 공식 C 적용

// 2단계: 기억 안에서 전파 소문 선택
var gossipToShareByA = _gossipEngine.SelectGossipToShare(agentA, agentB, topKForA);
var gossipToShareByB = _gossipEngine.SelectGossipToShare(agentB, agentA, topKForB);
```

> [!NOTE]
> Top-K 연산을 대화 루프 밖으로 추출하는 이유:
> 대화 진행 중(4턴) 소문 상태는 변하지 않으므로(사후 처리에서만 갱신), 루프 밖에서 1회 계산하여 재사용하는 것이 더 효율적이고 일관적입니다.

**변경 3: Top-K 점수 계산을 재사용 가능한 private 메서드로 추출**

`DialogueOrchestrator` 클래스 내에 아래의 헬퍼 메서드를 추가합니다:

```csharp
private List<KnownGossip> ComputeTopKGossips(AgentInstance speaker, AgentInstance listener, int k = 3)
{
    return speaker.KnownGossips.Values
        .Select(kg => {
            double score = kg.SubjectiveBelief;
            if (kg.Gossip.Subject == listener.AgentId) score += 0.15;
            if (!kg.HasSharedWithOthers) score += 0.25;
            if (speaker.RelationshipMap.TryGetValue(kg.Gossip.Subject, out var subjectRel))
                score += Math.Min(Math.Abs(subjectRel.Liking) / 250.0, 0.4);
            score += Math.Min(kg.Gossip.MutationCount * 0.08, 0.25);
            double secondsSinceAcquired = (DateTime.UtcNow - kg.AcquiredAt).TotalSeconds;
            score += Math.Max(0.0, (1.0 - secondsSinceAcquired / 1000.0) * 0.2);
            return new { Gossip = kg, Score = score };
        })
        .OrderByDescending(x => x.Score)
        .Take(k)
        .Select(x => x.Gossip)
        .ToList();
}
```
