## Must Analysis

#### 직관과 목적

- **핵심 아이디어**: 어떤 프로그램 지점에 도달하는 “모든 실행 경로”에서 참인 사실만 보존한다.
- **목표 성격**: “항상 안전함”을 증명한다. 하나의 경로라도 깨지면 그 사실은 유지되지 않는다.
- **전형적 용도**: Definitely initialized variables, Available expressions, Very-busy expressions, Must-alias 등 “보장”이 필요한 정적 보증.

---

#### 격자와 Meet 연산

- **격자 원소**: “반드시 참”인 사실들의 집합.
- **순서 관계**: 보통 “포함관계의 역순”을 쓴다. 사실이 많을수록 더 강한(위험한) 주장이라 위에 놓인다.
- **Meet(합류점)**:
    - **정의**:  
        [ in[n] ;=; \bigcap_{p \in pred(n)} out[p] ]
    - **의미**: 모든 경로에서 공통으로 유지되는 사실만 남긴다(교집합).

---

#### 방향, 초기값, 수렴

- **전방 분석(Forward)**: 들어오는 사실을 합치고 노드의 전이함수로 갱신.
    - **일반식**:  
        [ out[n] ;=; f_n(in[n]) ]
- **초기값(Entry)**: 전방 Must는 보통 (\top)에서 시작해 교집합으로 “깎아내린다”. 단, 실제 문제에 따라 Entry는 특수 취급하는 경우가 있다(예: Available expressions는 Entry에서 (\varnothing)을 둘 때가 많다).
- **종료 보장**: 유한 격자 + 단조 전이함수이면 고정점 반복으로 수렴한다.

---

#### 정확성(보수성)과 에러 성향

- **보수성 방향**: “항상 참”을 과감히 단정하지 않는다. 즉, Must 분석은 “거짓 양성(false positive)” 없이 사실을 증명하려 한다.
- **한계(정밀도 손실)**: 경로 비민감(path-insensitive) 합류 때문에 실제로는 항상 참인 사실을 증명하지 못하는 “거짓 음성(false negative)”이 생길 수 있다.

---

#### 작동 예시 1 — Definitely initialized

```c
int x;
if (cond) { x = 1; }
else      { x = 2; }
y = x;  // 여기서 x는 반드시 초기화됨(Must 참)
```

- **분기 양쪽 모두** x를 정의하므로 합류점에서  
    [ init_must = init_{\text{then}} \cap init_{\text{else}} = {x} \cap {x} = {x} ]
- 따라서 y = x 지점에서 “x는 반드시 초기화”가 유지된다.

```c
int x;
if (cond) { x = 1; }
y = x;  // 여기서는 한 경로에서 초기화가 없으므로 Must가 깨진다
```

- **else 경로**에서 초기화가 없으므로 교집합 결과에 x가 사라진다.

---

#### 작동 예시 2 — Available expressions(전방 Must)

```c
// 가정: a, b, c는 메모리 쓰기를 유발하지 않는 순수 표현
t1 = a + b;
if (cond) { t2 = a + b; }
else      { t3 = c + 1; }
t4 = a + b;  // 여기서 a+b는 '항상' 이전에 계산되어 사용 가능?
```

- **의도**: 합류점에서 a+b가 모든 경로에서 재계산 없이 사용 가능해야 “available”로 본다.
- **합류 결과**: then 경로는 a+b가 분기 직전 이미 계산됨, else 경로에서도 분기 전 계산이 있어야 합류에서 보존된다. 분기 전 t1 = a + b가 존재하므로  
    [ avail = avail_{\text{then}} \cap avail_{\text{else}} \ni (a+b) ]
- 단, 중간에 a 또는 b를 “변경(kill)”하는 문장이 있으면 해당 표현은 제거된다.

## May Analysis

#### 직관과 목적

- **핵심 아이디어**: “어떤 실행 경로에서라도” 참이면 보존한다.
- **목표 성격**: “가능성”을 놓치지 않는다. 하나의 경로라도 성립하면 유지한다.
- **전형적 용도**: Reaching definitions, Live variables(후방), May-alias, Points-to 등 “발생 가능성”이 중요한 문제.

---

#### 격자와 Meet 연산

- **격자 원소**: “참일 수도 있는” 사실들의 집합.
- **순서 관계**: 보통 표준 포함관계를 쓴다. 가능성을 더 많이 담을수록 위에 놓인다.
- **Meet(합류점)**:
    - **정의**:  
        [ in[n] ;=; \bigcup_{p \in pred(n)} out[p] ]
    - **의미**: 경로 중 하나라도 가지는 사실이면 합류에서 유지(합집합).

---

#### 방향, 초기값, 수렴

- **전방 분석(Forward)**:  
    [ out[n] ;=; f_n(in[n]) ]
- **초기값(Entry)**: 전방 May는 보통 (\bot = \varnothing)에서 시작해 합집합으로 “쌓아 올린다”.
- **종료 보장**: 유한 격자 + 단조 전이함수이면 고정점 반복으로 수렴한다.

---

#### 정확성(보수성)과 에러 성향

- **보수성 방향**: “가능성”을 놓치지 않는다. 즉, 실제로 도달 가능한 사실을 빼먹는 “거짓 음성(false negative)”을 피하는 쪽으로 동작한다.
- **한계(정밀도 손실)**: 실제로는 불가능한 사실도 합류에서 살아남아 “거짓 양성(false positive)”이 생길 수 있다(대표적으로 Reaching definitions가 불필요하게 많은 정의를 보고).

---

#### 작동 예시 1 — Reaching definitions(전방 May)

```c
x = 1;
if (cond) { x = 2; }
y = x;  // y에 도달 가능한 x의 정의는?
```

- **정의 집합**: 분기 전 {x←1}, then에서 {x←2}, else에서는 {x←1} 유지.
- **합류점**에서  
    [ reach(x) = {x\leftarrow 1} \cup {x\leftarrow 2} = {x\leftarrow 1,; x\leftarrow 2} ]
- 따라서 y에는 두 정의가 “도달 가능”하다고 본다.

---

#### 작동 예시 2 — Live variables(후방 May)

```c
int x = 0, z = 0;
if (cond) { z = x + 1; }
else      { /* z를 사용 안 함 */ }
return z;
```

- **후방 전파**: 사용(use) 지점에서 거꾸로 전파하며 합류에서 합집합.
- **해석**: 어떤 경로(cond=true)에서는 x가 z 계산에 쓰이므로, 분기 시작점에서 x는 “사용될 가능성이 있음(라이브)”. [ live_in = live_{\text{then}} \cup live_{\text{else}} \ni x ]

---

## 언제 어떤 걸 쓰나

- **안전 보장(위반이 하나라도 있으면 안 됨)**:
    
    - **Must**를 선호.
    - **예**: “변수는 이 지점 전 반드시 초기화되어야 한다”, “이 표현식은 항상 재계산 없이 사용 가능해야 한다.”
- **가능성 탐지(놓치면 안 됨)**:
    
    - **May**를 선호.
    - **예**: “이 정의가 여기까지 올 수 있는가”, “이 변수는 나중에 사용될 수 있는가(라이브)”, “이 포인터가 그 객체를 가리킬 수 있는가.”
- **정밀도 팁**:
    
    - **경로 민감도(Path-sensitivity)**를 올리면 Must/May 모두 정밀해지지만 비용 증가.
    - **컨텍스트 민감도(함수 호출 구분)**, **메모리 정밀화(에일리어싱 축소)**, **SSA 형태** 적용은 May의 거짓 양성과 Must의 거짓 음성 모두를 줄이는 데 유용.
    - **엔트리 초기값**은 문제 정의와 격자 설계에 따라 특수 처리할 수 있다(예: Available expressions의 Entry는 실무에서 (\varnothing)로 두는 관례가 흔함).
