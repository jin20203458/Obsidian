[품질 체크리스트]
1. LLVM/Clang 코딩 스타일 준수 (clang-format 기준)
2. API 사용 시 최신 LLVM/Clang API 문서 기준으로 유효성 확인
3. Null 포인터, 경계값, 예외 상황 처리 포함
4. Static Analyzer Checker: 경로 민감도(Path-sensitive) 고려, 상태 전이(State transitions) 명확히 정의
5. Clang-Tidy Check: AST 매칭 패턴 정확성, FixItHint 적용 시 안전성 검증

8. 빌드 및 실행 시 경고/에러 0개

1) 요구사항 분석 및 제약 조건 요약 (3~5줄)
2) AST/CFG 탐색 전략 또는 매칭 패턴 설계
3) 상태 전이 다이어그램 또는 로직 플로우 설계

5) 단위 테스트 작성 및 실행 결과 검증

[출력 형식]

지금부터 내가 제공한 Clang-Tidy c++소스코드에 대한 분석을 시작하자. Clang 버전은 19.x기준으로

### 분석 규칙
1. **분석 시작 전**  
   - 해당 코드가 속한 **함수의 개요(역할)**, **반환값**, **인자(매개변수)**를 먼저 설명해줘.  
   - 함수가 여러 개라면, 각 함수별로 위 내용을 정리해줘.
   - C++ 코드 색상 출력 버전으로 분석해줘 

2. **Line by Line 분석**  
   - 내가 붙여넣은 코드의 각 줄을 순서대로 분석해줘.  
   - 각 줄의 구성요소(변수, 상수, 연산자, 키워드 등)를 설명하고,  
     해당 줄에 **함수가 호출**되면 그 함수의 역할과 동작을 간단하게 설명해줘.  
   - 함수가 표준 라이브러리나 Clang-Tidy 내부 함수라면, 간단한 예시를 들어 직관적으로 이해할 수 있도록 해줘.
    - C++ 코드 색상 출력 버전으로 분석해줘 

3. **예시 제공**  
   - 내용이 직관적으로 이해하기 어렵거나 추상적이면,  
     간단한 **코드 예시** 또는 **비유**를 추가해서 설명해줘.

4. **출력 형식**  
   - **[함수 개요]** → 함수 설명, 반환값, 인자  
   - **[Line X]** → 해당 줄의 코드와 상세 분석  
   - **[예시]** → 필요 시 추가 예시

5. **목표**  
   - Clang-Tidy 소스코드 구조와 동작을 쉽게 이해할 수 있도록,  
     초보자도 읽고 바로 감이 올 수 있는 설명을 제공해줘.

해당 룰을 체크하는 Clang Static Analyzer (CSA) Checker를 제작

[개발 환경 및 타겟]
 - llvm 19.x 기준
 - 대상은 C, C++
   
[작업 프로세스 및 요구사항]
- 기획안 선 제출 (코드 작성 금지): 코드를 당장 만들지 말고, 먼저 어떻게 개발할지 기획안을 제출하고 나의 승인을 받은 후 코드를 제작할 것. `checkPreStmt` / `checkPostCall` 등 어떤 콜백을 사용할지, 상태 저장용 `REGISTER_MAP_WITH_PROGRAMSTATE` 활용 여부 등
    
- 오탐(FP) 최소화: 흐름 제어(Control Flow)나 제약 조건(Constraints) 처리를 통해 오탐을 어떻게 배제할지 구체적으로 명시할 것.
    
- 메시지 규칙: 외부 툴의 범용 매핑을 위해, 오류 메시지는 CWE나 CERT 같은 특정 보안 전문 용어를 배제하고 평이하고 이해하기 쉬운 한글로 작성할 것.
    
- 프로덕션 레벨 품질: 코드는 llvm 19.x 기준 가장 최신 방식을 사용하며, 실제 프로덕션 환경에 투입할 수 있도록 예외 처리 등을 포함해 최대한 무결하게 작성할 것.
