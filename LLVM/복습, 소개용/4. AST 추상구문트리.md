[[ASTContext]] : 자세한 내용 참조

**AST**는 Clang 파서가 소스 코드를 분석하여, 코드의 **문법적 구조와 의미를 계층적인 트리(tree) 형태로 표현한 자료구조**입니다. 단순히 텍스트였던 코드를 컴파일러가 이해하고 분석할 수 있는 구조적인 형태로 변환한 중간 결과물이죠.

![[Pasted image 20250909130520.png]]

> 이때 LLVM IR이랑 착각하면 안됩니다. 프론트엔드 계층의 중간 언어정도의 뜻으로 이해해야합니다.
> 실제 LLVM IR 은 언어 독립적으로 동작하지만, Clang AST 는 Clang에 종속적인 위치입니다.



## 왜 AST로 변환해야 할까?


### 왜 텍스트 그대로는 안 될까?

사람은 `a + b * c` 라는 텍스트를 보고 곱셈을 먼저 해야 한다는 것을 자연스럽게 알지만, 컴파일러는 단순히 글자의 나열으로 보기 때문에 연산의 우선순위나 코드의 포함 관계를 즉시 알 수 없습니다.

```scss
    +
   / \
  a   *
     / \
    b   c
```
핵심은 **연산자 우선순위**에 따라 코드의 논리적 구조를 시각적으로 나타내는 것입니다.
컴파일러는 이 트리 구조를 위에서부터 순회하며 "덧셈을 하려면 `a`와 `*`의 결과가 필요하구나. `*`의 결과를 얻으려면 `b`와 `c`를 먼저 곱해야겠군." 과 같은 순서로 코드의 실행 계획을 세우는 것입니다.

Clang은 여기에 'BinaryOperator'나 'DeclRefExpr' 같은 구체적인 명찰과 타입 정보를 붙여서 더 정교하게 관리하는 것이고요.

---

Clang-tidy에서 주로 사용하는 린터(AST 매쳐기반) 방식의 체커들은 대부분 AST를 이용합니다.
해당 방식은 조금더 하위 방식으로 3종류 정도가 더 있는데 

1. AST 매쳐 방식(가장 기본적)
 - 원하는 AST 노드들을 미리 입력해두고 AST를 전체 순회하면서 해당 노드들에 대한 정보를 수집
 - 해당 노드에 대하여 검증을 수행하는 방식
```
1. Clang이 소스코드 전체를 파싱하여 완전한 AST 생성
2. AST 매처가 AST 전체를 순회(traverse)하면서 패턴 매칭
3. 매칭된 노드들을 수집하여 결과 세트 생성  
4. 각 매칭된 노드에 대해 check() 함수 호출
```
[[NoAutoTypeCheck]]


2. RecursiveASTVisitor를 사용하는 방식
 - 직접 재귀적으로  AST 순회에 끼어들어서 본인이 오버라이드한 Visit 상황에서 개입하여 검증을 수행하는 방식
```
1. AST가 생성되면 즉시 순회 시작
2. 각 노드를 방문할 때마다 해당 Visit*** 메서드 호출
3. 실시간으로 분석 로직 실행
4. 순회 중 언제든 중단 가능 (return false)
```
[[NoShadowingCheck]]

3. CFG(Control Flow Graph) 기반 분석 방식 
```scss
Block 1 (Entry) → Block 2 (x > 0)
                     ↙         ↘
            Block 3 (x++)   Block 4 (x--)
                     ↘         ↙
                  Block 5 (return 0)
```
- 경로가 여러갈래로 갈라지는 경우 모든 경로에서의 확인이 필요할떄 CFG방식을 사용합니다. 
- Symbolic Excution과 같은 완전한 경로민감분석 보다는 제어흐름정보가 필요할떄 앞서 2가지 방식과 혼용하는 방식을 자주 사용합니다.
[[InitBeforeUseCheck]]
