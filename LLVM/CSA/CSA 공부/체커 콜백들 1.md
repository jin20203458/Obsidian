## 1. `check::ASTDecl<DECL>`

```cpp
template <typename DECL>
class ASTDecl {
  template <typename CHECKER>
  static void _checkDecl(void *checker, const Decl *D,
                         AnalysisManager& mgr, BugReporter &BR) {
    ((const CHECKER *)checker)->checkASTDecl(cast<DECL>(D), mgr, BR);
  }

  static bool _handlesDecl(const Decl *D) {
    return isa<DECL>(D);
  }
public:
  template <typename CHECKER>
  static void _register(CHECKER *checker, CheckerManager &mgr) {
    mgr._registerForDecl(
      CheckerManager::CheckDeclFunc(checker, _checkDecl<CHECKER>),
      _handlesDecl);
  }
};
```

- **“경로 탐색(ExprEngine) 들어가기 전에” AST를 한 번 훑는 단계**에서,  
    `DECL` 타입의 선언을 만날 때마다 호출될 수 있는 훅이야.
    
- 예를 들어 `ASTDecl<FunctionDecl>` 이면, TU 안의 모든 함수 선언에 대해 한 번씩.

## 4. `check::PreStmt<STMT>`

```cpp
template <typename STMT>
class PreStmt {
  template <typename CHECKER>
  static void _checkStmt(void *checker, const Stmt *S, CheckerContext &C) {
    ((const CHECKER *)checker)->checkPreStmt(cast<STMT>(S), C);
  }

  static bool _handlesStmt(const Stmt *S) {
    return isa<STMT>(S);
  }
public:
  template <typename CHECKER>
  static void _register(CHECKER *checker, CheckerManager &mgr) {
    mgr._registerForPreStmt(
      CheckerManager::CheckStmtFunc(checker, _checkStmt<CHECKER>),
      _handlesStmt);
  }
};
```

- “해당 STMT 타입의 문장을 **실행하기 직전**에 호출되는 콜백”을 등록하는 태그.
    
- 예: `PreStmt<BinaryOperator>`, `PreStmt<IfStmt>`, `PreStmt<CallExpr>` 등으로 사용.

- **명령이 실제로 “실행되기 직전”에 값을 보고 싶을 때 사용**

## 5. `check::PostStmt<STMT>`

```cpp
template <typename STMT>
class PostStmt {
  template <typename CHECKER>
  static void _checkStmt(void *checker, const Stmt *S, CheckerContext &C) {
    ((const CHECKER *)checker)->checkPostStmt(cast<STMT>(S), C);
  }

  static bool _handlesStmt(const Stmt *S) {
    return isa<STMT>(S);
  }
public:
  template <typename CHECKER>
  static void _register(CHECKER *checker, CheckerManager &mgr) {
    mgr._registerForPostStmt(
      CheckerManager::CheckStmtFunc(checker, _checkStmt<CHECKER>),
      _handlesStmt);
  }
};
```

- “해당 STMT 타입의 문장을 **실행한 직후**에 호출되는 콜백”.
- “실행 결과를 본 뒤” 상태를 기반으로 검사:

Pre와 Post의 구분 감각:

- Pre: “이걸 하기 **전에** 이 값/조건을 보고 싶다”
- Post: “이걸 하고 **난 후** 최종 SVal/State를 보고 싶다”

## 9. `check::PreCall`

```cpp
class PreCall {
  template <typename CHECKER>
  static void _checkCall(void *checker, const CallEvent &msg,
                         CheckerContext &C) {
    ((const CHECKER *)checker)->checkPreCall(msg, C);
  }

public:
  template <typename CHECKER>
  static void _register(CHECKER *checker, CheckerManager &mgr) {
    mgr._registerForPreCall(
      CheckerManager::CheckCallFunc(checker, _checkCall<CHECKER>));
  }
};
```

- 함수 호출/메서드 호출/연산자 호출 등,  
    “**CallEvent**로 추상화된 모든 호출”에 대해 **호출 직전**에 동작하는 콜백.
    
- `PreStmt<CallExpr>`와는 다르게:
    
    - AST `CallExpr`에 직접 묶이는 게 아니라
        
    - `CallEvent`라는 추상 계층(virtual function call, operator call, function pointer call 등 공통)을 통해 다룸.

### 언제 호출되나?

- ExprEngine에서 호출을 모델링할 때:
    
    1. `PreCall` 콜백들 호출 (`runCheckersForPreCall`)
        
    2. 실제 CallEvent 실행(요약/모델 적용)
        
    3. `PostCall` 콜백들 호출 (`runCheckersForPostCall`)

    2. 실제 호출 효과(상태 변화, 리턴값 SVal 생성 등) 적용
        
    3. **PostCall 콜백 호출** (`runCheckersForPostCall`)

## 11. `check::Location`

```cpp
class Location {
  template <typename CHECKER>
  static void _checkLocation(void *checker, SVal location, bool isLoad,
                             const Stmt *S, CheckerContext &C) {
    ((const CHECKER *)checker)->checkLocation(location, isLoad, S, C);
  }

public:
  template <typename CHECKER>
  static void _register(CHECKER *checker, CheckerManager &mgr) {
    mgr._registerForLocation(
      CheckerManager::CheckLocationFunc(checker, _checkLocation<CHECKER>));
  }
};
```

- **메모리 위치(location)에 접근(로드 또는 스토어)할 때마다** 호출되는 콜백.
- 매우 저수준이고 빈번하게 호출될 수 있는 훅.

- **인자 설명:**
    
    - `SVal Loc`: 접근하려는 메모리 위치(주소)
        
    - `bool isLoad`:
        
        - `true` → 로드(읽기)
            
        - `false` → 스토어(쓰기)
            
    - `const Stmt *S`: 해당 접근을 발생시킨 AST 노드
        

### 언제 호출되나?

- ExprEngine에서 메모리 읽기/쓰기 모델링할 때:
    
    - 변수 참조, 배열/포인터 역참조, 필드 접근 등
        
    - 실제로 SVal을 메모리에서 로드/스토어하기 직전에
        
    - `runCheckersForLocation`
        

### 어디에 쓰면 좋은가?

- **메모리 안전성 검사의 핵심**:
    
    - **널 포인터 역참조** 탐지  
        → `Loc`이 null/undefined일 때 에러
        
    - **범위 밖 메모리 접근**  
        → 배열 인덱스 초과, 해제된 메모리 접근
        
    - **초기화되지 않은 메모리 읽기**

## 12. `check::Bind`

```cpp
class Bind {
  template <typename CHECKER>
  static void _checkBind(void *checker, SVal location, SVal val, const Stmt *S,
                         CheckerContext &C) {
    ((const CHECKER *)checker)->checkBind(location, val, S, C);
  }

public:
  template <typename CHECKER>
  static void _register(CHECKER *checker, CheckerManager &mgr) {
    mgr._registerForBind(
      CheckerManager::CheckBindFunc(checker, _checkBind<CHECKER>));
  }
};
```
- **어떤 location에 어떤 값(val)이 바인딩(대입/저장)될 때** 호출.
- 메모리 상태 변화의 가장 기본 단위 추적.

- **인자:**
    
    - `SVal Loc`: 값이 저장될 메모리 위치
        
    - `SVal Val`: 저장되는 값
        
    - `const Stmt *S`: 이 바인딩을 유발한 AST 노드(대입문 등)
        

### 언제 호출되나?

- ExprEngine에서 변수 대입, 초기화, 필드 갱신 등  
    "메모리에 값을 쓰는" 모든 작업을 모델링할 때:
    
    - `runCheckersForBind`
        

### 어디에 쓰면 좋은가?

- **값 전파 및 상태 추적**:
    
    - **Taint 분석**: 오염된 값이 어디로 흘러가는지 추적  
        → `Val`이 tainted이면 `Loc`도 taint 전파
        
    - **특정 타입/범위 제약**: 특정 필드에 금지된 값 대입 검출
        
    - **메모리 할당 상태 갱신**: 포인터에 새로운 주소 대입 시 소유권/lifetime 기록
