## 21. `check::PointerEscape`

```cpp
class PointerEscape {
  template <typename CHECKER>
  static ProgramStateRef
  _checkPointerEscape(void *Checker,
                      ProgramStateRef State,
                      const InvalidatedSymbols &Escaped,
                      const CallEvent *Call,
                      PointerEscapeKind Kind,
                      RegionAndSymbolInvalidationTraits *ETraits) {

    if (!ETraits)
      return ((const CHECKER *)Checker)->checkPointerEscape(State,
                                                            Escaped,
                                                            Call,
                                                            Kind);

    InvalidatedSymbols RegularEscape;
    for (SymbolRef Sym : Escaped)
      if (!ETraits->hasTrait(
              Sym, RegionAndSymbolInvalidationTraits::TK_PreserveContents) &&
          !ETraits->hasTrait(
              Sym, RegionAndSymbolInvalidationTraits::TK_SuppressEscape))
        RegularEscape.insert(Sym);

    if (RegularEscape.empty())
      return State;

    return ((const CHECKER *)Checker)->checkPointerEscape(State,
                                                          RegularEscape,
                                                          Call,
                                                          Kind);
  }

public:
  template <typename CHECKER>
  static void _register(CHECKER *checker, CheckerManager &mgr) {
    mgr._registerForPointerEscape(
      CheckerManager::CheckPointerEscapeFunc(checker,
                                             _checkPointerEscape<CHECKER>));
  }
};
```
- **포인터가 현재 분석 context 밖으로 “escape” 하는 상황**을 알려주는 콜백.
    
- Escape란:
    
    - 글로벌 변수에 저장
    - 함수 밖으로 반환
    - 다른 스레드/콜백/컨테이너에 넘겨져 더 이상 안전하게 추적할 수 없는 상태  
      같은 걸 의미.

- 인자:
    
    - `State`: 현재 상태
        
    - `Escaped`: escape한 심볼 집합
        
    - `Call`: 이 escape가 특정 함수 호출과 관련된다면 그 CallEvent (없을 수도 있음)
        
    - `Kind`: escape 종류(예: 반환, 전역 저장, 인자 전달 등)
        

### 언제 호출되나?

- ExprEngine이 어떤 연산/함수 호출로 인해:
    
    - 특정 포인터/심볼이 “현재 함수/분석 영역 밖 객체에게 전달되었다”고 판단할 때
        
    - 예: `global = p;`, `return p;`, `store_in_container(p);` 등
        
- 그 시점에 `runCheckersForPointerEscape` 가 호출되고,  
    `PointerEscape` 콜백이 불림.
    

### 어디에 쓰면 좋은가?

- **ownership/lifetime 모델링**:
    
    - 어떤 포인터가 escape하면 “이 객체는 더 이상 이 함수 내에서만 관리되지 않는다”는 의미.
        
    - leak이나 double free를 검사할 때:
        
        - escape한 포인터는 더 이상 “이 함수 안에서 해제 안 했다고 leak”으로 보기 어려움  
            (다른 쪽에서 관리할 수 있으니까)
            
- 예:
    
    - “포인터가 escape 했으면, 이 체커는 그 심볼의 ownership 추적을 중단”
    - 또는 “escape 한 포인터를 특별 상태로 마킹”

## 22. `check::ConstPointerEscape`

```CPP
class ConstPointerEscape {
  template <typename CHECKER>
  static ProgramStateRef
  _checkConstPointerEscape(void *Checker,
                           ProgramStateRef State,
                           const InvalidatedSymbols &Escaped,
                           const CallEvent *Call,
                           PointerEscapeKind Kind,
                           RegionAndSymbolInvalidationTraits *ETraits) {

    if (!ETraits)
      return State;

    InvalidatedSymbols ConstEscape;
    for (SymbolRef Sym : Escaped) {
      if (ETraits->hasTrait(
              Sym, RegionAndSymbolInvalidationTraits::TK_PreserveContents) &&
          !ETraits->hasTrait(
              Sym, RegionAndSymbolInvalidationTraits::TK_SuppressEscape))
        ConstEscape.insert(Sym);
    }

    if (ConstEscape.empty())
      return State;

    return ((const CHECKER *)Checker)->checkConstPointerEscape(State,
                                                               ConstEscape,
                                                               Call,
                                                               Kind);
  }

public:
  template <typename CHECKER>
  static void _register(CHECKER *checker, CheckerManager &mgr) {
    mgr._registerForPointerEscape(
      CheckerManager::CheckPointerEscapeFunc(checker,
                                             _checkConstPointerEscape<CHECKER>));
  }
};
```

- `PointerEscape`의 특수 버전으로,  
    **“내용(contents)은 보존되는 const 포인터 escape”** 를 따로 처리하고 싶을 때 쓰는 훅.
    
- 예: `const T*` 또는 “읽기 전용으로만 넘겨지는 포인터”가 escape할 때,  
    일반 escape와는 다르게 취급하고 싶은 경우.

- `Escaped`: 이번에 “const escape”로 분류된 심볼들의 집합
    
- `Call`: 이 escape가 특정 함수 호출과 관련 있으면 그 CallEvent (없을 수도 있음)
    
- `Kind`: escape 종류 (반환, 글로벌로 저장 등)
    

### 내부 동작 포인트

- `_checkConstPointerEscape` 안에서:
    
    - `Escaped` 전체에서
        
        - `TK_PreserveContents` trait가 붙어 있고
            
        - `TK_SuppressEscape` 가 아닌 심볼만  
            → `ConstEscape`에 넣음.
            
    - `ConstEscape`가 비어 있지 않으면  
        → `CHECKER::checkConstPointerEscape` 호출.
        

즉:

- `PointerEscape`가 “일반적인 escape”를 다루는 반면,
    
- `ConstPointerEscape`는 “**내용을 보존해야 하는 const 성질의 escape 심볼만 따로 모아서**” 체커에게 넘겨주는 역할.

## 23. `check::Event<EVENT>`

```cpp
template <typename EVENT>
class Event {
  template <typename CHECKER>
  static void _checkEvent(void *checker, const void *event) {
    ((const CHECKER *)checker)->checkEvent(*(const EVENT *)event);
  }
public:
  template <typename CHECKER>
  static void _register(CHECKER *checker, CheckerManager &mgr) {
    mgr._registerListenerForEvent<EVENT>(
      CheckerManager::CheckEventFunc(checker, _checkEvent<CHECKER>));
  }
};
```
- **체커 간 혹은 엔진 내부의 “커스텀 이벤트 시스템”**을 위한 훅.
    
- `EVENT` 타입으로 정의된 “이벤트 구조체/클래스”가 발생하면,  
    그 이벤트를 구독(listen)하는 체커들이 `checkEvent(const EVENT &)` 콜백을 받는 구조.
### 언제 호출되나?

- 다른 코드(예: 다른 체커, ExprEngine, 어떤 유틸리티)에서
    
    - `CheckerManager::_dispatchEvent(E)` 같은 메서드를 호출할 때.
        
- `EventDispatcher<EVENT>` 와 같이 쓰면:
    
    - 특정 체커가 이벤트를 발생시키고,
        
    - 다른 체커들이 그 이벤트를 받는 **publish/subscribe 스타일** 통신 가능.
        

### 어디에 쓰면 좋은가?

- **체커 간 협업**:
    
    - 한 체커가 “어떤 특수 상황이 발생했다”는 이벤트를 보내고,
        
    - 다른 체커는 그걸 받아서 추가 분석/리포트 수행.

## 24. `eval::Assume`

```cpp
namespace eval {

class Assume {
  template <typename CHECKER>
  static ProgramStateRef _evalAssume(void *checker, ProgramStateRef state,
                                     SVal cond, bool assumption) {
    return ((const CHECKER *)checker)->evalAssume(state, cond, assumption);
  }

public:
  template <typename CHECKER>
  static void _register(CHECKER *checker, CheckerManager &mgr) {
    mgr._registerForEvalAssume(
      CheckerManager::EvalAssumeFunc(checker, _evalAssume<CHECKER>));
  }
};

} // namespace eval
```
- **제약(assumption) 처리를 커스터마이징**하고 싶을 때 쓰는 콜백.
    
- CSA는 분기/조건을 만날 때마다:
    
    - “이 경로에서는 cond가 true 라고 가정한다”
        
    - “이 경로에서는 cond가 false 라고 가정한다”  
        같은 식으로 ConstraintManager에 `assume`을 건다.
        
- `eval::Assume`는 바로 이 **assume 단계에 체커가 끼어들 수 있는 훅**이다.
### 언제 호출되나?

- ExprEngine이 분기/assume를 처리할 때:
    
    - 예: `if (x > 0)` 같은 조건
        
    - true 경로: Cond를 “> 0” 로 assume
        
    - false 경로: Cond를 “<= 0” 로 assume
        
- 이때 `runCheckersForEvalAssume` 를 통해  
    등록된 `eval::Assume` 체커들의 `evalAssume`가 호출된다.[](https://clang-analyzer.llvm.org/checker_dev_manual.html)​
    

### 어디에 쓰면 좋은가?

- **특수 도메인/추상 값에 대한 제약 관리**:
    
    - 예: 비트마스크 도메인, 문자열 길이, 범위 제한 등
        
    - 특정 심볼에 대해 “이 값은 항상 even/odd” 같은 강한 invariant를 유지하고 싶을 때
        
- 일반적인 체커는 보통 기본 ConstraintManager에 맡기고,  
    아주 특수한 추상 도메인을 다루는 체커가 이 훅을 활용한다.

## 25. `eval::Call`

```cpp
namespace eval {

class Call {
  template <typename CHECKER>
  static bool _evalCall(void *checker, const CallEvent &Call,
                        CheckerContext &C) {
    return ((const CHECKER *)checker)->evalCall(Call, C);
  }

public:
  template <typename CHECKER>
  static void _register(CHECKER *checker, CheckerManager &mgr) {
    mgr._registerForEvalCall(
      CheckerManager::EvalCallFunc(checker, _evalCall<CHECKER>));
  }
};

} // namespace eval
```
- **함수/메서드 호출의 “의미 자체”를 체커가 직접 평가(override)하고 싶을 때** 쓰는 콜백이야.[](https://clang-analyzer.llvm.org/checker_dev_manual.html)​
    
- `PreCall`/`PostCall` 이 “호출 전후에 훅을 거는 것”이라면,  
    `eval::Call` 은 **“이 호출이 상태를 어떻게 바꿀지”를 체커가 직접 정의**하는 훅이라고 보면 된다.[](https://clang.llvm.org/doxygen/classclang_1_1ento_1_1CheckerManager.html)​

- `CallEvent` 안에:
    
    - 호출 대상(함수/메서드/연산자)
        
    - 인자 SVal 목록
        
    - 리턴 타입 등 정보가 들어 있음.
        
- 리턴값 `bool` 의미:
    
    - `true` → 이 체커가 **이 호출을 완전히 처리했다** (엔진 기본 처리를 더 하지 말라)
        
    - `false` → 기본 엔진/다른 체커들이 **추가로 이 호출을 처리해도 된다**
        

### 언제 호출되나?

- ExprEngine이 함수/메서드 호출을 만날 때:
    
    - `runCheckersForEvalCall` 을 통해 등록된 `eval::Call` 체커들의 `evalCall`을 차례로 호출.[](https://clang-analyzer.llvm.org/checker_dev_manual.html)​
        
- 어떤 체커가 `true`를 리턴하면:
    
    - “이 호출은 내가 모델링 끝냈다”는 의미라  
        → 나머지 기본 모델은 건너뛰고 그 체커가 만든 상태로 진행.

## 26. `Checker` 템플릿 / `CheckerBase`

```cpp
class CheckerBase : public ProgramPointTag {
  CheckerNameRef Name;
  friend class ::clang::ento::CheckerManager;

public:
  StringRef getTagDescription() const override;
  CheckerNameRef getCheckerName() const;

  virtual void printState(raw_ostream &Out, ProgramStateRef State,
                          const char *NL, const char *Sep) const { }
};
```
- 모든 체커의 공통 베이스.
    
- `ProgramPointTag` 를 상속해서,  
    디버그/추적용 태그에 체커 이름을 쓸 수 있게 해준다.[](https://clang.llvm.org/doxygen/functions_func_r.html)​
    
- `printState` 를 오버라이드하면  
    `-analyzer-checker-option` 같은 디버그 옵션에서  
    ProgramState 내용 덤프를 커스터마이징 가능.[](https://clang.llvm.org/doxygen/classclang_1_1ento_1_1CheckerManager.html)​

## `Checker` variadic 템플릿

```cpp
template <typename CHECK1, typename... CHECKs>
class Checker : public CHECK1, public CHECKs..., public CheckerBase {
public:
  template <typename CHECKER>
  static void _register(CHECKER *checker, CheckerManager &mgr) {
    CHECK1::_register(checker, mgr);
    Checker<CHECKs...>::_register(checker, mgr);
  }
};

template <typename CHECK1>
class Checker<CHECK1> : public CHECK1, public CheckerBase {
public:
  template <typename CHECKER>
  static void _register(CHECKER *checker, CheckerManager &mgr) {
    CHECK1::_register(checker, mgr);
  }
};
```
- **여러 콜백 역할(check::PreStmt, check::DeadSymbols, eval::Call 등)을 한 클래스에 합쳐주는 믹스인 베이스**.[](https://clang-analyzer.llvm.org/checker_dev_manual.html)​
    
- `class MyChecker : public Checker<check::PreStmt<BinaryOperator>, check::DeadSymbols>`  
    이런 식으로 쓰면:
    
    - 이 클래스는 `check::PreStmt<BinaryOperator>` 와 `check::DeadSymbols` 두 역할과  
        `CheckerBase` 를 동시에 상속.[](https://clang-analyzer.llvm.org/checker_dev_manual.html)​
        
    - 등록 시 `_register`가 재귀적으로 두 역할의 `_register`를 모두 호출해서  
        → PreStmt/DeadSymbols 두 콜백에 다 등록됨.

## 27. `EventDispatcher<EVENT>`

```cpp
template <typename EVENT>
class EventDispatcher {
  CheckerManager *Mgr = nullptr;
public:
  EventDispatcher() = default;

  template <typename CHECKER>
  static void _register(CHECKER *checker, CheckerManager &mgr) {
    mgr._registerDispatcherForEvent<EVENT>();
    static_cast<EventDispatcher<EVENT> *>(checker)->Mgr = &mgr;
  }

  void dispatchEvent(const EVENT &event) const {
    Mgr->_dispatchEvent(event);
  }
};
```
- 아까 본 `check::Event<EVENT>` 가 “이벤트를 **받는** 쪽이라면”,  
    `EventDispatcher<EVENT>` 는 “이벤트를 **발생시키는** 쪽”을 담당하는 믹스인.[](https://clang.llvm.org/doxygen/classclang_1_1ento_1_1CheckerManager.html)​
    
- 한 체커가 동시에:
    
    - `Checker<check::Event<MyEvent>, EventDispatcher<MyEvent>>`  
        를 상속하면,
        
    - `checkEvent(const MyEvent &)` 로 이벤트를 받고,
        
    - `dispatchEvent(const MyEvent &E)` 로 이벤트를 보낼 수 있음.
        

## 언제 쓰나?

- 매우 고급/특수한 경우:
    
    - 복수의 체커가 서로 정보를 교환해야 할 때
        
    - 예:
        
        - A 체커: “어떤 함수 호출에서 특수 조건 발견” → 이벤트 발생
            
        - B 체커: 이 이벤트를 받아 추가 분석/리포트
            

일반적인 단일 체커 개발에서는 굳이 건드리지 않아도 된다.

## 전체 그림 정리

이제 `Checker.h` 안에서:

- **어디서 어떤 콜백이 정의되는지** (`check::PreStmt`, `check::Location`, `check::DeadSymbols`, `eval::Call`, …)
    
- **이 콜백들이 언제 호출되는지**
    
    - AST 레벨 (`ASTDecl`, `ASTCodeBody`, `EndOfTranslationUnit`)
        
    - ExprEngine 경로 레벨 (`PreStmt`, `Location`, `Bind`, `BeginFunction`, `EndFunction`, `DeadSymbols`, …)
        
- **체커 클래스가 이 콜백들을 어떻게 묶는지**
    
    - `Checker<역할...>` variadic 템플릿
        
- **고급 기능**
    
    - `eval::Assume` / `eval::Call` (제약/호출 의미 override)
        
    - `PointerEscape` / `ConstPointerEscape` (escape 관리)
        
    - `Event` / `EventDispatcher` (체커 간 이벤트 시스템)
        

까지 한 바퀴를 다 돌았어.
