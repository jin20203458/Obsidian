네, 드디어 컴파일러의 마지막 여정, **코드 생성 (Code Generation)** 단계입니다.

최적화 단계를 거쳐 '가장 효율적인' 형태가 된 **중간 표현(IR)**을 입력받아, 드디어 실제 CPU(하드웨어)가 직접 실행할 수 있는 **타겟 기계어(Target Machine Code)** 또는 **어셈블리 코드(Assembly Code)**로 번역합니다.

### 🎯 코드 생성의 목표

- **입력 (Input):** 최적화된 중간 표현 (IR)
    
- **출력 (Output):** 타겟 기계 코드 또는 어셈블리 코드
    
- **핵심 아이디어:** 기계와 독립적이었던(machine-independent) IR을, 특정 CPU(Intel x86, ARM 등)에 종속적인(machine-dependent) 코드로 변환합니다.
    

---

### 🤔 코드 생성은 왜 어려운가? (핵심 과제 3가지)

IR을 기계어로 바꾸는 것은 단순한 1:1 번역이 아닙니다. IR은 "무한한 임시 변수(`t1, t2...`)"와 "단순한 명령어"를 가정했지만, 실제 CPU는 "유한한 개수의 레지스터"와 "복잡하고 기묘한 명령어"를 가졌기 때문입니다.

이 과정은 다음 세 가지 핵심 문제를 해결해야 합니다.

#### 1. 명령어 선택 (Instruction Selection)

- **문제:** IR의 한 줄(예: `t1 = a + b`)을 어떤 기계어 명령어로 번역해야 가장 효율적일까요?
    
- **예시:** `x = y + 10` 이라는 IR이 있을 때,
    
    - (A) `MOV EAX, [y]` (y를 레지스터 EAX로)
        
    - (B) `ADD EAX, 10` (EAX에 10을 더함)
        
    - (C) `MOV [x], EAX` (EAX의 값을 x로)
        
    - 3개의 명령어로 번역할 수 있습니다.
        
- **복잡성:** 하지만 만약 CPU가 `INC` (1 증가) 명령어, 또는 `LEA` (메모리 주소 계산) 명령어처럼 덧셈을 더 효율적으로 처리하는 특별한 명령어를 가졌다면 그것을 선택해야 합니다. "가장 빠르고 짧은" 명령어 조합을 찾는 것이 핵심입니다.
    

#### 2. 레지스터 할당 (Register Allocation)

- **문제:** **코드 생성에서 가장 어렵고 중요한 문제입니다.**
    
- **상황:**
    
    - **IR:** `t1, t2, ..., t100` 등 무한한 수의 임시 변수 사용 가능.
        
    - **CPU:** `EAX, EBX, ECX...` 등 8개 또는 16개 남짓의 **유한한** 레지스터만 사용 가능.
        
- **비유:**
    
    - 레지스터는 "요리사의 작업대" (매우 빠름).
        
    - 메모리는 "냉장고" (느림).
        
- **목표:**
    
    - 가장 자주 사용되는 변수(데이터)들을 '작업대'(레지스터) 위에 올려두어야 합니다.
        
    - 하지만 작업대가 꽉 차면(레지스터 부족), 어쩔 수 없이 일부 데이터를 '냉장고'(메모리)에 잠시 저장했다가(`Spilling`), 필요할 때 다시 꺼내와야 합니다.
        
- **핵심:** 이 "어떤 변수를 레지스터에 남기고, 어떤 변수를 메모리로 쫓아낼지(Spill) 결정하는" 알고리즘이 프로그램의 전체 속도를 좌우합니다. (이론적으로 '그래프 색칠 문제(Graph Coloring Problem)'라는 NP-Complete 문제와 관련이 깊습니다.)
    

#### 3. 명령어 스케줄링 (Instruction Scheduling)

- **문제:** 현대 CPU는 여러 명령어를 동시에 처리합니다 (파이프라이닝). 이때, 어떤 명령어는 실행 시간이 오래 걸리고(예: 나눗셈, 메모리 읽기), 어떤 명령어는 금방 끝납니다(예: 덧셈).
    
- **목표:** CPU가 '놀지 않도록' 명령어의 순서를 재조정합니다.
    
- **예시:**
    
    1. `t1 = (엄청 오래 걸리는 계산)`
        
    2. `t2 = t1 + 1` (1번이 끝나야만 시작 가능 → CPU가 1번을 기다리며 '멈춤')
        
    3. `a = b + c` (1, 2번과 상관없는 계산)
        
- **최적화 (스케줄링):**
    
    1. `t1 = (엄청 오래 걸리는 계산)`
        
    2. `a = b + c` (1번이 계산되는 '동안' 3번을 먼저 실행!)
        
    3. `t2 = t1 + 1`
        
- 이렇게 순서를 바꾸어 CPU 파이프라인의 '공백'을 메워 성능을 향상시킵니다.

### 🎬 최종 산출물

이 모든 과정을 거치면, 드디어 `.s` (어셈블리 파일) 또는 `.o` / `.obj` (오브젝트 파일, 기계어) 파일이 생성됩니다.

이 파일은 아직 완전한 실행 파일(`exe`)은 아니며, `printf` 같은 외부 라이브러리 함수나 다른 코드 파일과 연결해주는 **링커(Linker)**라는 별도의 프로그램을 거쳐야 비로소 우리가 실행할 수 있는 최종 프로그램이 됩니다. (링커는 엄밀히 말해 컴파일러의 일부는 아니지만, 컴파일 과정의 마지막 파트너입니다.)

### 📚 컴파일러 이론 전체 요약

우리는 다음 6단계를 모두 여행했습니다.

|**단계**|**입력**|**작업**|**출력**|**핵심 이론/도구**|
|---|---|---|---|---|
|**1. 어휘 분석**|소스 코드|문자열 → 토큰|토큰 스트림|정규 표현식, 유한 오토마타|
|**2. 구문 분석**|토큰 스트림|문법 구조 검사|AST|문맥 자유 문법(CFG), 푸시다운 오토마타|
|**3. 의미 분석**|AST|타입 검사, 범위 검사|Annotated AST|타입 시스템, 심볼 테이블|
|**4. 중간 코드**|Annotated AST|AST → 선형화/평탄화|IR, CFG|삼주소 코드|
|**5. 최적화**|IR, CFG|IR 효율화 (속도/크기)|최적화된 IR|데이터 흐름 분석 (Liveness 등)|
|**6. 코드 생성**|최적화된 IR|IR → 기계어|어셈블리/기계어|레지스터 할당, 명령어 선택|

컴파일러 이론의 전체적인 그림이 그려지셨기를 바랍니다!
