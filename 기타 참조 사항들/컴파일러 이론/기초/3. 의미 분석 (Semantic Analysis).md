네, 문법적으로 완벽한 **추상 구문 트리(AST)**가 완성되었습니다. 하지만 문법에 맞다고 해서 다 올바른 프로그램은 아니죠.

예를 들어, "사과가 빨리 달린다"라는 문장은 문법적(주어+부사+동사)으로는 맞지만 '의미상' 말이 되지 않습니다.

컴파일러의 세 번째 단계인 **의미 분석 (Semantic Analysis)**은 바로 이 '의미'를 검사하는 과정입니다.

### 🎯 의미 분석의 목표

- **입력 (Input):** 구문 분석기가 만든 추상 구문 트리 (AST)
    
- **출력 (Output):** 의미 검사가 완료되고, '타입' 등의 추가 정보가 덧붙여진 **주석 달린 AST (Annotated AST)** (또는 의미상 오류 목록)
    
- **핵심 질문:** "이 코드가 문법은 맞는데, **말이 되는 소리**인가?"
    

---

### 1. 📚 핵심 이론: 타입 시스템 (Type System)

의미 분석의 _가장_ 중요한 작업은 **타입 검사(Type Checking)**입니다.

- **정적 타입 (Static Typing) 언어** (C, Java, C++, Rust, Go...)는 컴파일 시점에 모든 변수와 표현식의 타입을 알아내고, 타입 규칙 위반을 찾아냅니다.
    
- **타입 규칙의 예:**
    
    - "정수(`int`)와 문자열(`string`)은 더할 수 없다."
        
    - "배열이 아닌 변수에 `[ ]` 인덱스를 사용할 수 없다."
        
    - "함수를 호출할 때, 정의된 인수의 개수와 타입이 일치해야 한다."
        

**작동 방식:** 파서가 만든 AST를 '순회(Traversal)'하면서 각 노드(Node)의 타입을 검사합니다. 예를 들어 `a = b + c` 라는 AST 노드가 있다면:

1. `b`의 타입은 무엇인가? (예: `int`)
    
2. `c`의 타입은 무엇인가? (예: `string`)
    
3. `+` 연산자가 `int`와 `string`을 더할 수 있도록 정의되어 있는가? (아니다!)
    
4. **"의미 오류(Semantic Error)! 타입 불일치!"** → 컴파일 중단.
    

---

### 2. 📇 핵심 자료구조: 심볼 테이블 (Symbol Table)

"타입을 검사하려면 `b`나 `c`가 무슨 타입인지 알아야 하잖아? 그 정보는 어디에 있지?"

바로 이 질문에 답하는 것이 컴파일러의 핵심 자료구조인 **심볼 테이블(Symbol Table)**입니다.

- **정의:** 프로그램에 등장하는 모든 **'이름(Identifier)'**에 대한 정보를 저장하는 거대한 사전(Dictionary) 또는 해시 테이블입니다.
    
- **저장 정보:**
    
    - **이름 (Name):** `score`, `myFunction`
        
    - **종류 (Kind):** 변수(variable), 함수(function), 클래스(class), 타입(type)...
        
    - **타입 (Type):** `int`, `float`, `string`, `(int, int) -> float` (함수 시그니처)
        
    - **범위 (Scope):** 이 이름이 유효한 코드 블록 (예: `main` 함수 내부, 전역)
        
    - (이후 단계) **메모리 주소**
        

**심볼 테이블의 작동 (AST 순회 중):**

1. **선언문 발견 (`int x = 10;`)**:
    
    - 현재 범위의 심볼 테이블에 `x`가 이미 선언되었는지 확인 (중복 선언 오류).
        
    - `"x"`라는 이름으로 `{ Kind: Variable, Type: int }` 정보를 테이블에 **저장**합니다.
        
2. **사용문 발견 (`y = x + 5;`)**:
    
    - `y`를 심볼 테이블에서 **검색(Lookup)**합니다. 없으면 "선언되지 않은 변수(Undeclared variable)" 오류.
        
    - `x`를 심볼 테이블에서 **검색**합니다. 찾았습니다! 타입은 `int`.
        
    - `x`의 타입(`int`)과 `5`의 타입(`int`)이 `+` 연산에 적합하므로 통과.
        
    - `y`의 타입(예: `int`)에 `int` 결과를 대입할 수 있는지 확인. 통과.
        

---

### 3. 🗺️ 기타 의미 검사

심볼 테이블과 타입 시스템을 이용해 다음과 같은 다양한 '의미' 검사를 수행합니다.

- **범위 검사 (Scope Checking):**
    
    - `{ int a = 10; } ... a = 20;` → 중괄호를 벗어난 `a`는 심볼 테이블(해당 범위)에서 사라졌으므로 "범위를 벗어남(Out of scope)" 오류.
        
- **제어 흐름 검사 (Control Flow Checking):**
    
    - `break`나 `continue` 문이 반복문(`for`, `while`) 바깥에서 사용되지는 않았는가?
        
    - 값을 반환하는 함수(`int func()`)가 모든 경로에서 `return` 문을 가지고 있는가?
        
- **접근 제어 검사 (Access Control):** (객체지향 언어)
    
    - `private`로 선언된 멤버 변수를 클래스 바깥에서 접근하려고 하지는 않는가?
        

### 요약 및 다음 단계

의미 분석까지 통과하면, 이 프로그램은 **'문법적으로도 올바르고, 의미상으로도 말이 되는'** 프로그램임이 보장됩니다. AST는 모든 타입 정보와 심볼 정보가 풍부하게 담긴 '주석 달린 AST'로 업그레이드됩니다.

이제 드디어 컴파일러의 **'프론트엔드(Frontend)'**가 끝났습니다.

코드를 '이해'하는 단계가 끝났으니, 이제 코드를 '변환'하고 '생성'하는 **'백엔드(Backend)'**로 넘어갈 차례입니다. 그 첫 번째 단계는 특정 기계(CPU)에 종속되지 않는 공통 언어로 바꾸는 **중간 코드 생성(Intermediate Code Generation)**입니다.
