네, 의미 분석까지 모두 통과한 '주석 달린 AST'를 받았습니다. 이제 컴파일러는 이 코드가 문법적으로, 그리고 의미적으로도 완벽히 올바르다는 것을 압니다.

이제부터는 코드를 '이해'하는 **프론트엔드(Frontend)**가 끝나고, 코드를 '변환'하는 **백엔드(Backend)**가 시작됩니다.

그 첫 관문은 **중간 코드 생성 (Intermediate Code Generation, ICG)**입니다. 이 단계에서 생성된 코드를 **중간 표현 (Intermediate Representation, IR)**이라고 부릅니다.

### 🎯 중간 코드 생성의 목표

- **입력 (Input):** 의미 정보가 주석처럼 달린 AST (Annotated AST)
    
- **출력 (Output):** 중간 표현 (IR) (예: 삼주소 코드)
    
- **핵심 아이디어:** 복잡한 AST의 '트리 구조'를 기계어와 유사하지만 특정 기계(CPU)에는 종속되지 않는, **단순하고 선형적인 '명령어 리스트'**로 '평탄화(flatten)'시킵니다.

### 1. 🤔 왜 굳이 중간 코드를 만드나요?

소스 코드(예: Java)를 바로 특정 기계어(예: Intel x86)로 번역하면 되지 않을까요? 중간 코드를 사용하는 이유는 크게 두 가지입니다.

#### A. 이식성 / 재사용성 (Portability)

만약 N개의 프로그래밍 언어(C, Java, Python...)와 M개의 CPU 아키텍처(Intel, ARM, RISC-V...)가 있다면:

- **IR이 없다면:** 모든 언어-기계 조합(N x M) 만큼의 컴파일러를 만들어야 합니다. (예: "Java-to-Intel", "Java-to-ARM", "Python-to-Intel", ...)
    
- **IR이 있다면:**
    
    1. 각 언어를 IR로 번역하는 **프론트엔드 (N개)**
        
    2. IR을 각 기계어로 번역하는 **백엔드 (M개)**
        
    
    - 총 **N + M** 개의 모듈만 만들면 됩니다. Java 프론트엔드 하나만 만들면, Intel, ARM 백엔드가 알아서 기계어로 번역해줍니다.
        

#### B. 쉬운 최적화 (Optimization)

- **AST**는 너무 복잡하고 언어 종속적이라 최적화하기 어렵습니다.
    
- **기계어**는 너무 저수준이고(레지스터, 메모리 주소) CPU마다 달라서 최적화하기 까다롭습니다.
    
- **IR**은 이 둘의 '중간'에 있습니다. 언어와 기계로부터 독립적이며, 구조가 단순하고 균일해서 수학적 분석과 최적화를 수행하기에 _가장 완벽한_ 형태입니다.

### 2. 📝 중간 코드의 대표적인 형태: 삼주소 코드 (Three-Address Code, TAC)

가장 고전적이고 유명한 IR 형태입니다. 이름처럼 **최대 3개의 주소(변수, 상수, 임시 변수)**만을 사용해 모든 연산을 표현합니다.

**형식:** `result = operand1 op operand2` (연산자(op)가 최대 1개)

**예시: `x = (a + b) * (c - d);`**

복잡한 AST 트리를 어떻게 TAC로 평탄화하는지 보세요.

1. `t1 = a + b` (임시 변수 `t1`에 `a+b` 저장)
    
2. `t2 = c - d` (임시 변수 `t2`에 `c-d` 저장)
    
3. `t3 = t1 * t2` (`t1`과 `t2`를 곱함)
    
4. `x = t3` (최종 결과를 `x`에 대입)
    

AST의 복잡한 트리 노드들이 이렇게 간단한 4줄의 선형 코드로 바뀌었습니다. 이 변환은 AST를 후위 순회(Post-order traversal)하며 쉽게 생성할 수 있습니다.

---

### 3. 🗺️ IR의 핵심 자료구조: 제어 흐름 그래프 (Control Flow Graph, CFG)

삼주소 코드(TAC)는 `if`문이나 `while`문 같은 '제어 흐름'을 표현하기 위해 `jump`나 `if_false_jump` 같은 명령어를 사용합니다.

```
L1:                 // for (i=0; i<10; i++)
    t1 = i < 10
    if_false_jump t1, L2  // 만약 t1이 false면 L2(루프 끝)로 점프
    ...
    (루프 본문)
    ...
    i = i + 1
    jump L1           // 다시 L1(루프 시작)로 점프
L2:                 // 루프 끝
```

이런 `jump` 명령어들을 따라가기 어렵기 때문에, 컴파일러는 이 IR을 **'제어 흐름 그래프(CFG)'**라는 그래프 구조로 다시 만듭니다.

1. **기본 블록 (Basic Block):** `jump` 명령어 없이 순차적으로 실행되는 명령어들의 묶음입니다.
    
2. **엣지 (Edge):** 블록과 블록 사이의 `jump`를 나타내는 화살표입니다.
    

이 **CFG**는 컴파일러 최적화 단계에서 _가장_ 기본이 되는 핵심 자료구조입니다. 컴파일러는 이 그래프를 분석하며 코드의 흐름을 파악하고 최적화를 수행합니다.

### 요약 및 다음 단계

중간 코드 생성(ICG)은 복잡한 AST를 단순한 IR(예: 삼주소 코드)로 변환하고, 이를 다시 CFG라는 그래프로 재구성하는 단계입니다.

이제 언어와 기계로부터 독립적인, 최적화를 위한 완벽한 재료(CFG)가 준비되었습니다. 이 재료를 가공하여 더 빠르고 효율적인 코드를 만드는 **최적화 (Optimization)** 단계로 넘어갈 차례입니다.
