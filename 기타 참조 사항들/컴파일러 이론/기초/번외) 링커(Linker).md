좋습니다. 컴파일러가 자신의 일을 마친 직후의 시점에서 시작하겠습니다.

컴파일러는 `main.c`, `math.c`, `utils.c` 같은 소스 코드 파일들을 _개별적으로_ 번역하여 `main.o` (또는 `.obj`), `math.o`, `utils.o` 같은 **오브젝트 파일(Object File)**을 생성했습니다.

하지만 이 오브젝트 파일들은 아직 실행할 수 없습니다.

- `main.o`는 `printf` 함수를 호출하지만, `printf`의 실제 코드는 `main.o` 안에 없습니다.
    
- `main.o`는 `math.o`에 정의된 `sum` 함수를 호출하지만, `main.o`는 `sum`이 어디 있는지 모릅니다.
    

이처럼 **여러 조각으로 나뉜 코드와 라이브러리들을 하나로 묶어** 완전한 실행 파일(`.exe` 또는 `a.out`)로 만들어주는 프로그램이 바로 **링커(Linker)**입니다.

---

### 🎯 링커의 목표

- **입력 (Input):** 하나 이상의 오브젝트 파일 (`.o`, `.obj`) 및 라이브러리 파일 (`.a`, `.lib`, `.so`, `.dll`)
    
- **출력 (Output):** 하나의 **실행 파일(Executable File)** 또는 **공유 라이브러리(Shared Library)**
    
- **핵심 비유:** 흩어져 있는 여러 개의 원고(오브젝트 파일)와 참고 문헌(라이브러리)을 모아, 목차와 페이지 번호를 완성하여 '한 권의 책'(실행 파일)을 엮어내는 편집자.
    

---

### 🛠️ 링커의 2가지 핵심 작업

링커는 이 "책을 엮는" 과정에서 크게 두 가지 중요한 일을 합니다.

#### 1. 심볼 해석 (Symbol Resolution)

이것이 링커의 가장 핵심적인 임무입니다.

- **심볼(Symbol)이란?** 코드가 다른 파일에서 참조할 수 있도록 외부에 공개한 '이름'입니다. (예: 함수 이름, 전역 변수 이름)
    
- **문제 상황:**
    
    - `main.c` (컴파일된 `main.o`)는 `sum`이라는 함수를 호출합니다. (`sum`은 "정의되지 않음")
        
    - `math.c` (컴파일된 `math.o`)는 `sum`이라는 함수를 정의합니다. (`sum`은 "정의됨")
        
- **링커의 작업:**
    
    1. 모든 오브젝트 파일을 스캔하여 "정의된 심볼"과 "정의되지 않은 심볼"의 목록을 만듭니다.
        
    2. `main.o`가 필요로 하는 "정의되지 않은" `sum`을 `math.o`의 "정의된" `sum`과 **연결(link)**시켜 줍니다.
        
    3. 마찬가지로, `main.o`가 필요로 하는 `printf`를 표준 라이브러리(`libc`) 파일에서 찾아 연결합니다.
        

> 💡 **"Undefined reference to..."** 또는 **"Unresolved external symbol..."** 프로그래밍하면서 한 번쯤 본 이 에러가 바로 링커가 심볼 해석에 실패했을 때(필요한 심볼을 어디에서도 찾지 못했을 때) 발생하는 **"링크 에러(Link Error)"**입니다.

#### 2. 재배치 (Relocation)

- **문제 상황:** 컴파일러는 `main.c`를 컴파일할 때, `main` 함수가 최종 실행 파일의 _어느 메모리 주소_에 위치할지 전혀 모릅니다. 그래서 일단 "0번지부터 시작한다"고 가정하고 코드를 생성합니다. (다른 `math.c`, `utils.c`도 마찬가지)
    
- **링커의 작업:**
    
    1. 최종 실행 파일의 **메모리 지도(Address Space)**를 만듭니다.
        
    2. "코드는 `0x1000`번지부터, 데이터는 `0x8000`번지부터 시작하자"고 결정합니다.
        
    3. `main.o`의 코드는 `0x1000`에, `math.o`의 코드는 `0x2000`에 배치합니다.
        
    4. 이제 1번 작업(심볼 해석)에서 찾은 `sum` 함수의 _최종 주소_가 `0x2000`임을 알게 되었습니다.
        
    5. `main.o` 코드 내의 `call sum` 명령어를 `call 0x2000` 과 같이 **실제 최종 주소로 수정(patch)**합니다. 이 과정을 **재배치**라고 합니다.

### 📦 정적 링킹 vs 동적 링킹

링커가 라이브러리를 연결하는 방식에는 두 가지가 있습니다.

- **정적 링킹 (Static Linking)**
    
    - `printf` 같은 라이브러리 코드를 '복사'해서 실행 파일 안에 _전부_ 집어넣습니다.
        
    - **장점:** 실행 파일 하나만 있으면 모든 환경에서 실행됩니다. (의존성 문제 없음)
        
    - **단점:** 실행 파일의 크기가 매우 커지고, 여러 프로그램이 동일한 `printf` 코드를 메모리에 중복해서 올리므로 비효율적입니다. (예: `.a`, `.lib` 파일 사용)
        
- **동적 링킹 (Dynamic Linking)**
    
    - 실행 파일에는 `printf` 코드 대신 "실행 시점에 `printf.dll` (또는 `printf.so`) 파일을 찾아 연결하라"는 '약속(stub)'만 남겨둡니다.
        
    - **장점:** 실행 파일이 작고, `printf` 코드를 시스템 메모리에 한 번만 올려두고 모든 프로그램이 '공유'하므로 효율적입니다.
        
    - **단점:** 해당 `.dll`이나 `.so` 파일이 없으면 프로그램이 실행되지 않습니다. (의존성 문제, 일명 "DLL Hell")

### 요약

**컴파일러**는 소스 코드를 기계어로 번역하는 **'번역가'**이고, **링커**는 번역된 여러 개의 기계어 조각과 라이브러리를 모아 실제 실행 가능한 프로그램으로 엮어내는 **'최종 편집자'**입니다.
