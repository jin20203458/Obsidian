### 컴파일러에서 Sema의 역할

컴파일러는 보통 다음과 같은 단계를 거칩니다.

1. **어휘 분석 (Lexing):** 코드를 단어(토큰) 단위로 쪼갭니다. (`int`, `a`, `=`, `"hello"`, `;`)
    
2. **구문 분석 (Parsing):** 이 토큰들이 C++ 문법에 맞게 잘 배열되었는지 확인합니다.
    
    - `int a = "hello";` -> 문법 구조(`타입 변수 = 값;`)는 일단 맞다고 봅니다.
        
    - 이 과정에서 **AST(추상 구문 트리)**가 생성됩니다.
        
3. **의미 분석 (Sema):** **(바로 여기입니다!)**
    
    - 구문 분석(Parsing)이 만들어낸 AST를 꼼꼼히 살펴보며 코드의 '의미'를 검증합니다.
        
    - `int a = "hello";` -> "어? `int`(정수) 타입 변수에 `"hello"`(문자열) 값을 넣으려고 하네? 이건 **의미상 오류**야!"
        

Sema 단계에서 주로 확인하는 것들은 다음과 같습니다.

- **타입 검사 (Type Checking):** 변수 타입이 호환되는지 (방금 본 예시)
    
- **선언 검사 (Declaration Checking):** `x = 10;` 코드를 만났을 때, `x`라는 변수가 이전에 선언된 적 있는지
    
- **유효 범위 검사 (Scope Checking):** 함수 안에서 선언된 지역 변수를 함수 밖에서 쓰려고 하지는 않는지
    
- **함수 호출 검사:** `MyFunction(10, 20);` 호출 시, `MyFunction`이 실제로 인자 2개를 받도록 정의되었는지

## AST와 Sema의 관계: 결과물과 작업자

- **AST**: `Sema`의 모든 분석 **결과가 명시적으로 기록된 데이터 구조(화석)**입니다. AST 자체는 `Sema` 객체를 알지 못합니다.
    
- **Sema**: AST라는 **결과물을 만드는 작업자(공룡)**입니다. `Sema`는 자신의 모든 분석 결과를 AST 노드의 종류, 속성, 구조 형태로 영구적으로 새겨 넣습니다.
    
- **`ASTConsumer`**: 완성된 **AST(화석)**만 보고도 코드의 모든 의미를 파악할 수 있는 **분석가(고생물학자)**입니다. `Sema`의 존재를 몰라도 분석이 가능합니다.

##  SemaConsumer의 존재 이유: 읽기를 넘어 "개입"하기

대부분의 분석은 `ASTConsumer`로 충분하지만, 다음과 같은 **고급 작업**을 위해서는 `Sema` 객체에 직접 접근할 수 있는 `SemaConsumer`가 필요합니다.

- **AST 수정 및 생성**: 리팩토링 도구처럼 코드 구조를 안전하게 변경하고 즉시 검증해야 할 때.
    
- **분석 과정 제어**: 필요에 따라 템플릿 인스턴스화를 강제로 유발해야 할 때.
    
- **코드 생성 개입**: 컴파일러 백엔드(CodeGen)와 더 긴밀하게 상호작용해야 할 때.

sema가 만든 결과물을 단순 분석하는게 아니라 실제 결과물을 수정및 개입할때 필요
