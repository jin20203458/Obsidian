
## 전통적인 컴파일러의 논리 흐름 (파이프라인 모델)

교과서적인 컴파일러는 각 단계가 독립적으로 순차 실행되는 '파이프라인(Pipeline)' 구조를 가집니다. 앞 단계의 출력이 다음 단계의 입력으로 그대로 전달되는 컨베이어 벨트와 같습니다.

**소스 코드: `int result = a + 10;`**

0. **전처리 확장**

1. **어휘 분석 (Lexical Analysis)**
    
    - **역할**: 소스 코드를 읽어 의미 있는 최소 단위인 **토큰(Token)**의 스트림으로 분해합니다.
        
    - **출력**: `[int]`, `[result]`, `[=]`, `[a]`, `[+]`, `[10]`, `[;]`
        
2. **구문 분석 (Syntax Analysis / Parsing)**
    
    - **역할**: 토큰 스트림을 받아 언어의 문법 규칙(Grammar)에 따라 **파스 트리(Parse Tree)** 또는 초기 **추상 구문 트리(AST)**를 만듭니다. 이 단계에서는 오직 문법 구조의 유효성만 검사합니다.
        
    - **결과 (AST)**:
        
        - `=` (할당 연산자)
            
            - 왼쪽: `int result` (변수 선언)
                
            - 오른쪽: `+` (덧셈 연산자)
                
                - 왼쪽: `a` (식별자)
                    
                - 오른쪽: `10` (정수 리터럴)
                    
    - **한계**: 이 단계의 AST는 `a`가 선언되었는지, `a`와 `10`을 더할 수 있는지 등 의미 정보는 전혀 모르는 순수한 구조 트리입니다.
        
3. **의미 분석 (Semantic Analysis)**
    
    - **역할**: 구문 분석 단계에서 생성된 **전체 AST**를 처음부터 순회하며 의미적 유효성을 검사하고, AST에 의미 정보를 추가(주석 달기)하거나 수정합니다.
        
    - **수행 작업**:
        
        - **기호 테이블(Symbol Table)**을 사용하여 `a`가 선언되었는지 확인합니다.
            
        - `a`의 타입과 `10`의 타입이 덧셈 연산에 유효한지 **타입 검사(Type Checking)**를 수행합니다.
            
        - 필요 시, 타입이 다르다면 암시적 형 변환 노드를 AST에 추가합니다.
            
    - **출력**: 의미 정보가 완전히 부여되고 검증된 **"의미 분석 완료 AST"**.
        
4. **중간 코드 생성 및 최적화 → ... → 최종 코드 생성**
    
    - 의미 분석이 끝난 AST를 기반으로 기계 독립적인 중간 표현(IR)을 생성하고, 최적화를 거쳐 최종 실행 코드를 만듭니다.

## Clang 컴파일러의 논리 흐름 (통합/협력 모델)

Clang은 구문 분석과 의미 분석이 분리되지 않고, 파서(Parser)와 의미 분석기(Sema)가 매우 긴밀하게 협력하여 AST를 함께 만들어 나가는 통합 모델을 사용합니다.

**소스 코드: `int result = a + 10;`**

1. **어휘 분석 (Lexical Analysis)**
    
    - **역할**: 전통적인 모델과 동일하게 소스 코드를 토큰 스트림으로 변환합니다. 전처리기(`Preprocessor`)가 이 과정에 깊이 관여하여 매크로 확장 등을 처리합니다.
        
    - **특징**: 파서의 요청이 있을 때마다 토큰을 하나씩 생성하여 전달하는 **요청 기반(On-demand)**으로 동작합니다.
        
2. **구문 분석 + 의미 분석 (Parsing + Semantic Analysis in collaboration)**
    
    - **역할**: 파서와 Sema가 실시간으로 대화하며 AST를 구축합니다.
        
    - **동작 흐름**:
        
        - **(1) 파서**: `Lexer`로부터 토큰들을 받으며 문법 구조를 인식합니다. `a + 10` 부분을 인식하고, 이것이 표현식임을 알아챕니다.
            
        - **(2) 파서 → Sema (위임)**: 파서는 AST 노드를 직접 만들지 않습니다. 대신, Sema에게 작업을 위임합니다. "Sema야, 식별자 `a`랑 정수 `10`으로 덧셈 표현식을 만들려고 해. 의미 분석하고, 해당 `BinaryOperator` AST 노드 좀 만들어줘."
            
        - **(3) Sema (분석 및 생성)**:
            
            - 기호 테이블을 참조하여 `a`가 선언되었는지, 타입은 무엇인지 확인합니다.
                
            - `a`의 타입과 `10`(int)의 타입이 덧셈에 유효한지 **타입 검사**를 수행합니다.
                
            - **의미 정보가 완전히 포함된** `BinaryOperator` 노드를 생성합니다. 이 노드 자체에 결과 타입이 무엇인지 등의 정보가 기록됩니다.
                
        - **(4) Sema → 파서 (반환)**: 생성된 `BinaryOperator` 노드를 파서에게 돌려줍니다.
            
        - **(5) 파서**: 이제 `int result = ...` 부분을 처리합니다. Sema로부터 받은 표현식 노드를 사용하여 변수 선언 구조를 인식합니다.
            
        - **(6) 파서 → Sema (위임)**: 다시 Sema에게 요청합니다. "Sema야, `result`라는 `int` 변수를 선언하고, 아까 받은 표현식 노드로 초기화하려고 해. 문제 없는지 보고, `VarDecl` 노드 만들어줘."
            
        - **(7) Sema (분석 및 생성)**: `result` 이름이 중복되지 않는지 확인하고, 초기화가 유효한지 검사한 후, **완성된 `VarDecl` 노드**를 생성합니다.
            
        - **(8) `HandleTopLevelDecl` 호출**: 이 `VarDecl`이 최상위 선언이라면, Sema는 이 완성된 AST "조각"을 `ASTConsumer`(예: Clang-Tidy)에게 `HandleTopLevelDecl` 콜백을 통해 전달합니다.
            
3. **최종 코드 생성 (LLVM IR)**
    
    - 이렇게 실시간으로 의미 분석까지 완료된 AST가 생성되면, 이를 기반으로 LLVM IR(중간 표현)을 생성하여 이후의 최적화 및 백엔드 처리를 진행합니다.
        

## 핵심 차이점 요약

|특징|전통적인 컴파일러|Clang 컴파일러|
|---|---|---|
|**구조**|**파이프라인 모델** (단계별 분리)|**통합 모델** (파서와 Sema의 협력)|
|**AST 생성 주체**|**파서**가 구조를 만들고, **Sema**가 나중에 정보를 추가|**Sema**가 파서의 요청에 따라 의미 분석 후 직접 생성|
|**의미 분석 시점**|구문 분석이 **끝난 후** 전체 트리를 대상으로 수행|구문 분석 **과정 중에** 실시간으로 함께 수행|
|**장점**|개념적으로 단순하고 각 단계가 명확함|오류 위치가 매우 정확하고, 실시간 코드 분석/툴링에 강력함|
|**비유**|**컨베이어 벨트 공장**: 각 공정(단계)이 독립적|**숙련공 팀**: 설계자(Sema)와 현장 작업자(파서)가 함께 소통하며 제작|
