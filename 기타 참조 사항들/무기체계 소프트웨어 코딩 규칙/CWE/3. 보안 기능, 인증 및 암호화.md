### **1) CWE-259: 하드코딩된 비밀번호 사용**

[[HardcodedPasswordCheck]]
[공용]
**규칙: 소스코드 내부에 비밀번호를 평문으로 기록하지 말고, 외부 설정 파일이나 환경 변수를 통해 관리해야 한다.**

#### ❌ 잘못된 예

```c
// ❌ 소스코드에 관리자 비밀번호가 노출됨
if (strcmp(password, "Admin_!@#$1234") == 0) {
    login_success();
}
```

#### ✅ 바른 예

```c
// ✅ 보안된 구성 파일이나 환경 변수에서 암호화된 비밀번호를 가져옴
char *saved_password = getenv("DB_PASSWORD"); 
if (compare_passwords(password, saved_password)) {
    login_success();
}
```

### **2) CWE-285: 부적절한 인가 (Improper Authorization)**

[[AuthorizationCheck]]

**규칙: 사용자가 요청한 기능을 수행하기 전에, 해당 기능을 실행할 권한이 있는지 반드시 확인해야 한다.**

#### ❌ 잘못된 예


```c
void delete_user(int user_id) {
    // ❌ 관리자 권한 확인 없이 삭제 로직 수행
    db_query("DELETE FROM users WHERE id = %d", user_id);
}
```

#### ✅ 바른 예


```c
void delete_user(int user_id, UserSession *session) {
    // ✅ 세션을 통해 요청자가 'ADMIN'인지 권한을 먼저 검사
    if (session->role != ROLE_ADMIN) {
        log_error("Unauthorized access attempt");
        return;
    }
    db_query("DELETE FROM users WHERE id = %d", user_id);
}
```

### **3) CWE-306: 중요 기능에 대한 인증 누락**

[[MissingAuthCheck]]

**규칙: 결제, 정보 수정 등 민감한 기능을 실행할 때는 사용자 재인증 절차를 거쳐야 한다.**

#### ❌ 잘못된 예


```c
void update_bank_account(int account_id, char *new_info) {
    // ❌ 로그인 세션만 있으면 추가 인증 없이 계좌 정보 변경 허용
    perform_update(account_id, new_info);
}
```

#### ✅ 바른 예


```c
void update_bank_account(int account_id, char *new_info) {
    // ✅ 중요 작업 전 패스워드 재입력 또는 2FA 인증 확인
    if (!verify_reauthentication()) {
        return;
    }
    perform_update(account_id, new_info);
}
```

### **4) CWE-307: 과도한 인증 시도 제한 부재**

[[BruteForcePreventionCheck]]

**규칙: 동일한 계정에 대해 반복적인 인증 실패가 발생할 경우, 계정 잠금 또는 지연 시간을 설정해야 한다.**

#### ❌ 잘못된 예


```c
while (login_failed) {
    // ❌ 실패 횟수 제한 없이 무한 반복 가능 (무차별 대입 공격에 취약)
    try_login(user_id, input_password);
}
```

#### ✅ 바른 예


```c
if (get_fail_count(user_id) >= 5) {
    // ✅ 5회 이상 실패 시 계정 일시 잠금 또는 CAPTCHA 요구
    lock_account(user_id);
    return ERROR_LOCKED;
}
```

### **5) CWE-312: 민감 정보의 평문 저장**

[[CleartextStorageCheck]]

**규칙: 비밀번호, 카드 정보 등 민감한 데이터는 반드시 암호화하여 저장해야 한다.**

#### ❌ 잘못된 예


```c
// ❌ 비밀번호를 텍스트 파일에 그대로 기록
fprintf(file, "ID:%s, PW:%s", id, password);
```

#### ✅ 바른 예


```c
// ✅ 해시 처리되거나 암호화된 데이터만 저장
char *hashed_pw = hash_password(password);
fprintf(file, "ID:%s, PW:%s", id, hashed_pw);
```

### **6) CWE-319: 민감 정보의 평문 전송**

[[CleartextTransmissionCheck]]

**규칙: 네트워크를 통해 민감한 정보를 전송할 때는 HTTPS, TLS 등 암호화된 통신 구간을 사용해야 한다.**

#### ❌ 잘못된 예


```c
// ❌ 일반 HTTP 포트(80)를 사용하여 데이터를 평문 전송
int sock = connect_to_server("example.com", 80);
send(sock, user_credentials, len, 0);
```

#### ✅ 바른 예


```c
// ✅ 암호화된 SSL/TLS 포트(443)와 라이브러리 사용
SSL *ssl = connect_to_server_secure("example.com", 443);
SSL_write(ssl, user_credentials, len);
```

### **7) CWE-321: 하드코딩된 암호화 키 사용**

[[HardcodedKeyCheck]]

**규칙: 암호화에 사용되는 키(Key)를 소스코드에 직접 입력하지 말고 외부에서 안전하게 주입받아야 한다.**

#### ❌ 잘못된 예


```c
// ❌ 소스코드에 대칭키가 그대로 노출됨
char *key = "MY_SECRET_AES_KEY_2024"; 
AES_encrypt(data, key, encrypted_data);
```

#### ✅ 바른 예


```c
// ✅ Key Management System(KMS)이나 보안 저장소에서 키를 로드
char *key = load_key_from_secure_vault();
AES_encrypt(data, key, encrypted_data);
```

### **8) CWE-327: 취약한 암호 알고리즘 사용**
[공통]
[[BrokenCryptoCheck]]

**규칙: MD5, SHA-1, DES 등 이미 취약점이 발견된 알고리즘 대신 AES-256, SHA-256 이상의 최신 알고리즘을 사용해야 한다.**

#### ❌ 잘못된 예


```c
// ❌ 보안성이 낮은 MD5 알고리즘 사용
MD5_Init(&ctx);
MD5_Update(&ctx, data, len);
```

#### ✅ 바른 예


```c
// ✅ 현재 표준인 SHA-256 사용
SHA256_Init(&ctx);
SHA256_Update(&ctx, data, len);
```

### **9) CWE-330: 불충분한 무작위 값 사용**
[공통]
[[InsufficientRandomCheck]]

**규칙: 보안과 관련된 난수(세션 ID, 암호화 키 등) 생성 시 예측 불가능한 암호학적 난수 생성기(CSPRNG)를 사용해야 한다.**

#### ❌ 잘못된 예


```c
// ❌ 시간 기반의 단순 rand() 함수는 예측이 가능함
srand(time(NULL));
int session_id = rand();
```

#### ✅ 바른 예


```c
// ✅ 운영체제가 제공하는 보안 난수 소스 사용
int session_id;
getrandom(&session_id, sizeof(session_id), 0);
```

### **10) CWE-494: 무결성 검사 없는 코드 다운로드**

[[IntegrityCheckCheck]]

**규칙: 외부에서 업데이트 파일이나 실행 코드를 다운로드할 때, 디지털 서명이나 해시값을 통해 변조 여부를 확인해야 한다.**

#### ❌ 잘못된 예


```c
void update() {
    download("update.bin");
    system("./update.bin"); // ❌ 변조 확인 없이 즉시 실행
}
```

#### ✅ 바른 예


```c
void update() {
    download("update.bin");
    if (verify_signature("update.bin", PUBLIC_KEY)) { // ✅ 디지털 서명 검증
        system("./update.bin");
    }
}
```

### **11) CWE-521: 취약한 패스워드 요구조건**

[[WeakPasswordPolicyCheck]]

**규칙: 최소 길이, 대소문자, 숫자, 특수문자 조합 등 충분히 복잡한 비밀번호 정책을 강제해야 한다.**

#### ❌ 잘못된 예


```c
if (strlen(new_password) < 4) { // ❌ 너무 짧은 비밀번호 허용
    return ERROR_WEAK;
}
```

#### ✅ 바른 예


```c
// ✅ 복잡도 체크 로직 포함 (길이 10자 이상, 특수문자 포함 등)
if (!is_strong_password(new_password)) {
    return ERROR_WEAK_POLICY;
}
```

### **12) CWE-732: 중요 자원에 대한 잘못된 권한 설정**

[[PermissionAssignmentCheck]]

**규칙: 시스템 파일이나 설정 파일을 생성할 때, 권한을 최소화하여 '누구나 수정'할 수 없도록 해야 한다.**

#### ❌ 잘못된 예


```c
// ❌ 파일 권한을 777(모든 사용자 읽기/쓰기/실행)로 설정
chmod("config.db", 0777);
```

#### ✅ 바른 예


```c
// ✅ 소유자만 읽고 쓸 수 있도록 최소 권한(600) 부여
chmod("config.db", 0600);
```

### **13) CWE-759: 솔트(Salt) 없는 해시 사용**

[공용]
[[NoSaltedHashCheck]]
**규칙: 비밀번호를 해싱할 때 사용자별로 고유한 Salt(무작위 문자열)를 추가하여 레인보우 테이블 공격을 방어해야 한다.**

#### ❌ 잘못된 예


```c
// ❌ 비밀번호만 그대로 해싱 (동일 비밀번호는 동일 해시값이 나옴)
char *hash = SHA256(password);
```

#### ✅ 바른 예


```c
// ✅ 사용자별 고유 Salt를 추가하여 해싱
char *salt = generate_unique_salt();
char *hash = SHA256(password + salt);
```
