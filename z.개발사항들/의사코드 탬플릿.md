Obsidian의 내장 LaTeX 렌더러(KaTeX)는 모든 LaTeX 패키지의 기능을 지원하지 않습니다. 따라서 **가장 기본적이고 호환성 높은 명령어만 사용**하는 것이 중요합니다.

## 1. `algorithmicx`의 확장 명령어 사용 금지

가장 흔한 오류의 원인입니다. `\Call`이나 `\Comment` 같은 명령어는 `algorithmicx` 패키지에서 제공하지만, KaTeX는 이를 지원하지 않습니다.

- **잘못된 예**﻿:
```
\State $q \gets$ \Call{Partition}{$A, p, r$}
\Comment{This is a comment}
```

- **올바른 예**﻿: 함수 호출이나 주석은 일반 텍스트로 직접 표현합니다.
```
  \State $q \gets \text{Partition(}A, p, r\text{)}$
\State \texttt{// This is a comment}
```

## 2. 줄 번호 옵션 `[1]` 사용 금지

`\begin{algorithmic}[1]`과 같이 줄 번호를 자동으로 매기는 옵션은 KaTeX에서 오류를 발생시킵니다.

- **잘못된 예**﻿: `\begin{algorithmic}[1]`
    
- **올바른 예**﻿: `\begin{algorithmic}`
    

## 3. 빈 줄을 위한 `\Statex` 사용 주의

`\Statex` 명령어는 프로시저 사이의 간격을 조절하기 위해 사용되곤 하지만, `\EndProcedure` 바로 뒤와 같이 특정 위치에서는 오류를 유발합니다. 프로시저 사이의 간격은 자연스럽게 두는 것이 가장 안전합니다.

- **잘못된 예**﻿:
    
    `\EndProcedure \Statex \Procedure{...}`

- **올바른 예**﻿: `\Statex` 없이 바로 다음 프로시저를 정의합니다.
    
    `\EndProcedure \Procedure{...}`


## 4. 특수 문자 이스케이프 처리

주석이나 텍스트 내에서 LaTeX의 특수 문자(`#`, `_` 등)를 사용할 때는 반드시 백슬래시(`\`)로 이스케이프 처리해야 합니다.

- **잘못된 예**﻿: `\texttt{// #include}`
    
- **올바른 예**﻿: `\texttt{// \#include}`

## 최종 템플릿 (복사하여 사용)

아래는 위 원칙들을 모두 준수하여 만든, Obsidian에서 안전하게 사용할 수 있는 최종 템플릿입니다.

```
\begin{algorithm}
\caption{알고리즘 이름}
\begin{algorithmic}
  \Procedure{첫 번째 함수 이름}{$매개변수1, 매개변수2$}
    \State $변수 \gets 초기값$
    \For{$i \gets 1$ \To $n-1$}
      \If{$A[i] > max$}
        \State $max \gets A[i]$
      \EndIf
    \EndFor
    \State \Return $max$
  \EndProcedure

  \Procedure{두 번째 함수 이름}{$매개변수$}
    \State \texttt{// 이곳에 로직을 작성합니다.}
    \State 다른\_함수\_호출()
    \If{조건문}
        \State \Return \textbf{true}
    \Else
        \State \Return \textbf{false}
    \EndIf
  \EndProcedure
\end{algorithmic}
\end{algorithm}

```

이 원칙과 템플릿을 따르시면, 앞으로 Obsidian의 LaTeX 환경에서 발생하는 대부분의 렌더링 오류를 피할 수 있습니다.
