## 1. Structured Prompting

### Principle
Using XML tags to physically segregate prompt components (instructions, data, examples, output format) increases LLM instruction-following rates by over 30%.

### Basis
- [Source] "XML Prompting as Grammar-Constrained Interaction" (arXiv:2509.08182, 2025.09)
  → Mathematically proves that XML-tag-based prompting operates as grammar-constrained interaction, offering convergence guarantees.
- [Source] Anthropic Official Documentation (2024–2025, regularly updated)
  → "XML tags help Claude parse complex prompts unambiguously."
- [Source] "The PICCO Framework" (arXiv:2604.14197, 2026)
  → Explicit separation framework: Persona, Instructions, Context, Constraints, Output.

### Practical Rules
- Use `<system_directive>` to define Roles and Tasks.
- Use `<rules>` or `<constraints>` to specify behavioral constraints and exclusions.
- Use `<context>` for reference data (e.g., world settings, prior narratives).
- Use `<output_format>` to define output schemas (JSON schema, tag structures, etc.).
- Use `<example>` for few-shot examples.
- Ensure formatting tags are consistent across a single prompt: choose either XML or Markdown, do not mix them.

## 2. Data-Instruction Separation

### Principle
Failing to isolate data (user inputs, external texts) from instructions (rules, commands) within a prompt can lead to data being misinterpreted as instructions, causing hallucinations or exposing prompt injection vulnerabilities.

### Basis
- [Source] "Defense Against Prompt Injection Attack" (ACL 2025, 44 citations)
  → Identified as OWASP LLM security threat #1. The root cause is the lack of separation between instructions and data.
- [Source] "How Not to Detect Prompt Injections with an LLM" (ACM AISec 2025 Workshop, 2025.10)
  → LLM-based injection detection (KAD) has inherent architectural vulnerabilities (e.g., DataFlip attacks can reduce detection rate to 0%). Architectural isolation remains the only fundamentally effective defense strategy.

### Practical Rules
- Always wrap user input data in `<user_input>` or `<context>` tags.
- Explicitly prevent the model from generating data outside its designated domain (e.g., the "Zero-Numeric" principle: "Do not include numerical statistics in narrative summaries").
- When passing data generated by one model into the instruction set of another model, always wrap it in dedicated data tags to distinguish it from the instructions.

## 3. Consumer-Driven Contract Design

### Principle
The output format of a prompt must be defined by the expectations of the consumer (the parsing code or subsequent agent in the pipeline). The objective is "parsable text", not just "eloquent text".

### Basis
- [Source] "Software Engineering for Prompt-Enabled Systems" (arXiv:2503.02400, 2025–2026)
  → Treats prompts as typed interfaces using contract-driven design.
- [Source] "JSONSchemaBench: A Rigorous Benchmark of Structured Outputs for Language Models" (arXiv:2501.10868, ICML 2025 ES-FoMo Workshop)
  → Outlines how prompting-only JSON generation fails significantly as schema complexity increases (failure rate ranges from ~14% for simple schemas to 91% for complex ones). While grammar-constrained decoding (e.g., Guidance, XGrammar) achieves nearly 100% accuracy on simple schemas, performance varies widely across frameworks for nested/recursive schemas.
- [Source] "Chain-of-Collaboration Prompting Framework" (arXiv, 2025.05)
  → Outputs exchanged between agents must be in parsable, validated formats.

### Practical Rules
- If the output is expected to be parsed as JSON:
  → Set `responseMimeType: "application/json"`.
  → Include an exact JSON schema example in the prompt.
  → Implement defensive parsing in code (e.g., `ExtractJson`, `DeserializeSafe`).
- If the output is to be parsed via specific tags:
  → Explicitly define target tags (e.g., `<plot>`, `<chronicle>`) in the prompt.
  → Implement fallback parsing logic (e.g., falling back to raw text if tags are missing).
- In multi-step pipelines:
  → Align the output specification of each stage with the expected input schema of the next stage.
  → Provide context to the agent about where and how its output will be consumed.

## 4. Numerical Safety Clamping

### Principle
LLMs exhibit systematic errors (2% to 15%) during arithmetic operations. Prompts dealing with numerical values must explicitly specify valid bounds (clamping) and error-handling rules.

### Basis
- [Source] "Steering Large Language Models between Code Execution and Textual Reasoning" (ICLR 2025, CodeSteer)
  → Demonstrates that LLMs often prefer textual reasoning over code execution, leading to arithmetic errors. Even when a task is 100% solvable via code execution (Code Interpreter), textual reasoning degrades performance, showing inverse scaling with model and task complexity.
- [Source] "Demystifying Errors in LLM Reasoning Traces: An Empirical Study of Code Execution Simulation" (ACM TOSEM, 2026 / arXiv:2512.00215, 2025.11)
  → Evaluated four SOTA reasoning models (Claude 4, DeepSeek R1, Gemini, GPT-4o). Out of nine error categories, computation error was the most frequent. Tool augmentation (calculator plugins) corrected 58% of computation errors.

### Practical Rules
- For bounded stats (e.g., "100/100"): Explicitly state that values cannot exceed `[0 - Max]` (Clamping).
- For single values (e.g., "Gold: 1000"): Specify if unbounded operations are allowed.
- Provide range guidelines: e.g., "Increase by 5-10", "Decrease by 20%".
- Explicitly state that unchanged attributes must retain their previous values.
- Where possible, delegate math/numerical calculations to application code rather than prompt-based inference (leveraging Tool Use).

## 5. System Instruction vs. User Prompt Placement Strategy

### Principle
Placing role definitions, behavioral constraints, output formats, and few-shot examples inside the System Instruction, and reserving the User Prompt only for the actual query/task, yields optimal performance.

### Basis
- [Source] "LLM Shots: Best Fired at System or User Prompts?" (Halil et al., 2025, Huawei)
  → Found that placing the core query in the User Prompt, and everything else in the System Prompt, yields optimal results.
- [Source] Google Official Documentation (Gemini 3, 2026.06)
  → "Place core behavioral constraints, role, and output format in the System Instruction or at the very top of the prompt."
- [Source] OpenAI Official Documentation (2026)
  → Recommended order for Developer/System messages: Identity → Instructions → Examples → Context.

### Practical Rules (GRC Implementation)
- Place in **System Instruction**:
  → GM Persona/Personality definition
  → World settings (`<master_setting>`)
  → Lorebook data
  → Formatting rules (`<syntax_rules>`)
- Place in **User Prompt**:
  → Mid/long-term narrative context (`<background_context>`)
  → Current user actions (`<current_action>`)
  → Final execution prompt (`<final_instruction>`)

## 6. Few-Shot Example Strategy

### Principle
Few-shot examples should always be included. The optimal count is 2 to 4 examples, with format consistency being the most critical factor. Clear examples can offset the need for extensive explicit instructions.

### Basis
- [Source] Google Official Documentation (2026)
  → "Always recommend including few-shot examples. Prompts without examples are less effective."
  → "If examples are clear enough, you may reduce explicit instructions."
- [Source] "LLM Shots" (2025)
  → 2–4 examples are optimal for most models.
- [Source] OpenAI Official Documentation (2026)
  → "Provide examples illustrating the range of inputs. Include edge cases."

### Practical Rules
- Number of examples: 2 to 4 (too many causes overfitting; too few yields negligible benefit).
- Format consistency: XML tags, indentation, and separators must be identical across all examples.
- Diversity: Include both happy paths and edge cases.
- Placement: Place inside the System Instruction (never in the User Prompt).
- GRC Status:
  → The chat view formatting rules contain 1 dialog/monologue example inside `<example>` tags. ✅
  → Agent prompts currently have no few-shot examples. ⚠️ (Opportunity for improvement)

## 7. Temperature and Generation Config Optimization

### Principle
For the Gemini 3.x model family, sampling parameters such as `temperature`, `top_p`, and `top_k` should not be explicitly set. Instead, omit them (or pass `null`) to let the model run on its API defaults (Default = 1.0).

### Basis
- [Source] Google Official Documentation (Gemini 3.5 Flash & 3.1 Pro Developer Guide, verified 2026.06)
  → ⚠️ "For Gemini 3.x models, we strongly recommend maintaining temperature, top_p, and top_k at their default values. Altering them may cause loop generation (repetitive outputs) or performance degradation."
  → Consistent structuring for tasks like precise JSON output or code generation should be controlled using System Instruction definitions and Response Schema (Structured Outputs), rather than artificially lowering the temperature.

### Temperature Guidelines (For reference in Gemini 2.x and below, or third-party models)
| Task Type            | Recommended Temperature |
|----------------------|------------------------|
| JSON / Structured    | 0.0 ~ 0.2              |
| Code Generation      | 0.0 ~ 0.3              |
| Classification / Ext.| 0.0 ~ 0.2              |
| Summarization        | 0.3 ~ 0.5              |
| General Q&A          | 0.5 ~ 0.7              |
| Creative / Narrative | 0.7 ~ 1.0              |
| Brainstorming        | 0.8 ~ 1.2              |

### Practical Rules (GRC Status)
- **Unified API Implementation**: Converted the `Temperature` property in `GenerationConfig` to a nullable type (`float?`) and set it to `null`. This ensures the parameter is omitted from the API request payload.
- **SessionArchitect Agent**: Omitted (runs on defaults). ✅
- **Status Update (StatusAPI)**: Omitted (though it outputs JSON, it has been changed from `0.6` to defaults to prevent infinite repetition loops). ✅
- **Mid/Long-Term Narrative Summarization & Japanese Translation**: Omitted (runs on defaults). ✅

## 8. Reasoning/Thinking Model Prompt Strategy

### Principle
When prompting reasoning-native models (e.g., Gemini 2.5/3, Claude Extended Thinking), instruct them on "WHAT" to do rather than "HOW" to do it. CoT directives are obsolete.

### Basis
- [Source] Google Official Documentation (Gemini Thinking, 2026.06.04)
  → "Gemini 2.5/3 series automatically generate internal reasoning thoughts, so demanding the model to explain reasoning steps in the output is generally unnecessary."
  → "For complex tasks, adding 'Think very hard before answering' can enhance performance."
- [Source] OpenAI Official Documentation (2026)
  → Reasoning models (o-series): No CoT instructions needed. A concise, direct statement of the goal is optimal. Place core instructions in Developer messages and delegate process details to the model.
  → Standard models (GPT-4o, etc.): Explicit, step-by-step instructions remain highly effective. Structured prompts are essential.

### Thinking Level Guidelines (Gemini 3)
| Task Complexity      | Recommended Thinking Level |
|----------------------|----------------------------|
| Simple (RAG, Classif)| minimal or OFF             |
| Medium (Comp, Inference) | low ~ medium          |
| Complex (Math, Code) | high                       |

### Practical Rules (GRC Implementation)
- Status Window Updates: `ThinkingLevel.minimal` ✅ (simple JSON mapping)
- Mid-Term Narrative Summarization: `ThinkingLevel.high` ✅ (complex narrative integration)
- Long-Term Memory Merging: `ThinkingLevel.medium` ✅ (medium-complexity compression)
- Do not use CoT instructions like "think step-by-step" (reasoning is built-in).

## 9. Prompt Length and Token Efficiency

### Principle
As prompt length increases, performance gains diminish rapidly. Place critical information at the beginning or end of the prompt, avoiding the middle.

### Basis
- [Source] "Lost in the Middle: How Language Models Use Long Contexts" (Liu et al., TACL 2024)
  → LLMs struggle to pay attention to information placed in the middle of long contexts. Information at the beginning and the end is retrieved most effectively, a pattern observed even in long-context models.
- [Source] "Incorporating Token Usage into Prompting Strategy Evaluation" (arXiv:2505.14880, 2025.05)
  → Token count increases result in diminishing performance returns. Increasing few-shot examples from 3 to 8 multiplies token costs by 10x while yielding negligible accuracy improvements. Proposes the "Big-Otok" framework for efficiency-aware prompt evaluation.

### Practical Rules
- Place core instructions at the very top of the prompt.
- Place massive context data in the middle, strictly isolated using XML tags.
- Place the final execution prompt at the very bottom (e.g., inside `<final_instruction>`).
- Use bridging transitions: Connect the context and the task explicitly (e.g., "Based on the information provided above...").
- Avoid prompt bloat: Concise, focused instructions are more effective than overly verbose explanations.

## 10. Negative Prompting

### Principle
Positive instructions ("Do X") are generally more reliable than negative ones ("Do not do Y"). However, negative constraints are effective for blocking default model behaviors.

### Basis
- Negative prompting requires the LLM to represent the forbidden action in its attention space before attempting to suppress it, which paradoxically increases the likelihood of generating the forbidden output.
- Google, OpenAI, and Anthropic recommend using negative constraints sparingly, and always pairing them with positive alternatives.

### Practical Rules
- Prioritize positive framing: e.g., instead of "Do not use bullet points," use "Use numbered paragraphs."
- Valid use cases for negative instructions:
  → Blocking default behaviors: "Do not use markdown formatting."
  → Safety/Domain boundaries: "Do not describe or narrate actions, dialogues, or emotions for the PC."
  → Format enforcement: "Do not output any conversational text outside of the JSON."
- Avoid vague negative guidelines: "Do not hallucinate" is ineffective. Provide concrete reference contexts instead.

## 11. Multilingual (Korean) Prompt Considerations

### Principle
Due to tokenization differences, CJK languages (Chinese, Japanese, Korean) consume 2 to 3 times more tokens than English for equivalent semantic content. Structural markers must be employed more explicitly.

### Basis
- [Source] "Multilingual Prompt Engineering in Large Language Models: A Survey Across NLP Tasks" (arXiv:2505.11665, 2025.05)
  → Surveyed 36 papers, 39 prompting techniques, and 250 languages. Explores the performance gap between high-resource languages (English, Chinese) and low-resource/non-English languages. Explores mitigation strategies like Prompt Adaptation, Translation-based prompting, and Cross-lingual few-shot prompting. Korean, as an SOV language, benefits from more explicit structural markers and prompt adaptation strategies.

### Practical Rules (GRC Implementation)
- System Instruction role/rules: Use Korean if the end-user interaction is in Korean. ✅
- Tags and separators: Maintain XML tags in English (e.g., `<rules>`, `<output_format>`). ✅
- Token efficiency: Minimize redundant phrasing to save tokens.
- Explicit language target: Always include explicit output instructions (e.g., "Write the response in Korean."). ✅

## 12. Self-Correction Pattern (Future Consideration)

### Principle
The "Generate → Evaluate → Refine" self-correction loop is one of the most common and robust agent chaining patterns recommended by Anthropic.

### Basis
- [Source] Anthropic Official Documentation (2024–2025)
  → "The most common chaining pattern is self-correction: generate a draft, evaluate it against criteria, and refine it based on evaluation feedback."

### GRC Implementation Potential
- An evaluation step can be inserted immediately after the final agent stage (e.g., `PromptGen`) to automatically verify that the generated system prompt does not conflict with the runtime environment.
- Current Status: Not implemented (reserved as a future optimization task).
