좋습니다. 컴파일러의 첫 번째 관문인 **어휘 분석(Lexical Analysis)**부터 자세히 살펴보겠습니다. 이 단계를 수행하는 프로그램을 **렉서(Lexer)** 또는 **스캐너(Scanner)**라고 부릅니다.

### 🎯 어휘 분석의 목표

어휘 분석의 핵심 목표는 간단합니다. 문자 단위로 이루어진 원시 소스 코드를 읽어들여, 문법적으로 의미 있는 **'토큰(Token)'**의 연속으로 변환하는 것입니다.

- **입력 (Input):** 소스 코드 문자열 (예: `if (score > 10)`)
    
- **출력 (Output):** 토큰의 스트림(Stream) (예: `IF`, `(`, `ID(score)`, `GT`, `NUM(10)`, `)`)
    

이 과정에서 프로그래밍 언어의 의미와 관련 없는 공백, 줄 바꿈, 주석 등은 모두 제거합니다.

### 1. 🤓 핵심 이론: 정규 언어와 오토마타

"어떻게 `if`는 키워드이고 `score`는 변수 이름(Identifier)이라고 컴퓨터가 인식할 수 있을까?" 이 질문에 대한 해답이 바로 형식 언어 이론에 있습니다.

#### A. 기술(Description): 정규 표현식 (Regular Expression)

먼저, 각 토큰이 어떤 '모양'인지 정의해야 합니다. 이때 사용되는 언어가 바로 **정규 표현식**입니다.

- **키워드 'if'**: `if`
    
- **숫자 (Number)**: `[0-9]+` (0~9의 숫자가 1번 이상 반복)
    
- **식별자 (Identifier)**: `[a-zA-Z_][a-zA-Z0-9_]*` (알파벳이나 밑줄로 시작하고, 이후에는 알파벳/숫자/밑줄이 0번 이상 반복)
    

이처럼 정규 표현식은 우리가 찾고자 하는 토큰의 '패턴'을 명시적으로 기술하는 도구입니다.

#### B. 인식(Recognition): 유한 오토마타 (Finite Automaton, FA)

정규 표현식으로 '정의'는 했지만, 이 패턴을 '인식'하는 기계가 필요합니다. 이 기계를 **유한 오토마타(Finite Automaton, FA)**라고 부릅니다.

유한 오토마타는 '상태(State)'와 '전이(Transition)'로 이루어진 간단한 추상 기계입니다.

- 현재 '상태'에서 '입력 문자'를 하나 읽습니다.
    
- '전이 규칙'에 따라 다음 '상태'로 이동합니다.
    
- 정해진 '종결 상태(Final State)'에 도달하면, "패턴을 인식했다!"고 선언합니다.
    

**예시: 식별자를 인식하는 유한 오토마타**

1. **Start (시작 상태):** 아무것도 읽지 않은 상태.
    
2. `[a-zA-Z_]` (문자나 `_`)를 읽으면 -> **S1 (인식 중) 상태**로 이동.
    
    - 만약 `[0-9]` (숫자)를 먼저 읽으면 -> **Error (오류) 상태**로 이동.
        
3. **S1 상태:**
    
    - `[a-zA-Z0-9_]` (문자/숫자/`_`)를 읽으면 -> 계속 **S1 상태**에 머무름 (루프).
        
    - 다른 문자(공백, `(`, `)` 등)를 읽으면 -> **Accept (종결 상태)**가 됩니다. "지금까지 읽은 `score`는 식별자가 맞다!"
        

실제 **렉서(Lexer)** 프로그램은 이 유한 오토마타를 코드로 구현한 것입니다.

---

### 2. 🛠 실제 구현: 렉서 생성기 (Lexer Generator)

이론적으로는 우리가 직접 유한 오토마타를 C나 Java 코드로 하나하나 구현할 수 있습니다. (예: 거대한 `switch` 문 사용)

하지만 매우 복잡하고 오류가 발생하기 쉽기 때문에, 보통은 **렉서 생성기(Lexer Generator)**라는 도구를 사용합니다.

- **대표적인 도구:** `Lex` (C언어용), `Flex` (Fast Lex), `JLex` (Java용)
    

**작동 방식:**

1. **개발자:** `.l` 파일에 정규 표현식(패턴)과, 해당 패턴이 인식되었을 때 실행할 코드(액션)를 정의합니다.
    
    - `"if"` `{ return TOKEN_IF; }`
        
    - `[a-zA-Z_][a-zA-Z0-9_]*` `{ return TOKEN_ID; }`
        
    - `[0-9]+` `{ return TOKEN_NUM; }`
        
2. **`Lex` 도구:** 이 `.l` 파일을 읽어서, 이 모든 규칙을 인식하는 **유한 오토마타(DFA)를 C 코드(`lex.yy.c`)로 자동 생성**해줍니다.
    
3. **컴파일러:** 우리는 이 자동 생성된 C 코드를 우리 컴파일러의 다른 부분과 함께 컴파일해서 사용합니다.

### 3. 🚦 어휘 분석의 주요 원칙 (모호성 해결)

만약 입력 코드가 `ifelse`라면, 렉서는 이것을 `if`와 `else`로 봐야 할까요, 아니면 `ifelse`라는 하나의 식별자로 봐야 할까요? 렉서는 다음과 같은 두 가지 핵심 원칙으로 이 모호성을 해결합니다.

1. **최장 일치 원칙 (Longest Match Rule)**
    
    - 렉서는 항상 입력 스트림에서 '가장 긴' 문자열과 일치하는 패턴을 선택합니다.
        
    - `ifelse`는 `if` 패턴(길이 2)보다 `[a-zA-Z_]...*` 패턴(길이 6)과 더 길게 일치하므로, `ifelse`는 하나의 `TOKEN_ID`가 됩니다.
        
    - `10.5`는 `10` (정수)이 아니라 `10.5` (실수)로 인식됩니다.
        
2. **우선순위 원칙 (Priority Rule)**
    
    - 만약 가장 긴 문자열이 두 개 이상의 패턴과 일치한다면 (예: `if`는 `if` 패턴과 `TOKEN_ID` 패턴 모두와 일치),
        
    - 렉서 생성기 규칙 파일(`.l` 파일)에 **먼저 정의된 규칙**이 우선순위를 갖습니다.
        
    - 이 때문에 보통 키워드 규칙(`if`, `while`, `for`)을 일반적인 식별자 규칙(`TOKEN_ID`)보다 _앞에_ 정의합니다.
        

### 요약 및 다음 단계

어휘 분석기가 이 모든 과정을 마치면, 소스 코드는 더 이상 단순한 문자열이 아니라 의미 있는 블록(토큰)의 '흐름(Stream)'이 됩니다.

`"if (score > 10)"` → `[IF, '(', ID(score), GT, NUM(10), ')']`

이 토큰 스트림은 이제 문장의 구조를 파악하는 **'구문 분석기(Parser)'**로 전달됩니다.
