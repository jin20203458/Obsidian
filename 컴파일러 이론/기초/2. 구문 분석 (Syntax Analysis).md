네, 바로 **구문 분석 (Syntax Analysis)** 단계로 넘어가겠습니다. 이 단계를 수행하는 프로그램을 **파서(Parser)**라고 부릅니다.

어휘 분석기가 코드를 의미 있는 단어(토큰)로 쪼갰다면, 구문 분석기는 이 단어(토큰)들이 모여 **올바른 문장 구조**를 이루는지 검사합니다.

- **어휘 분석 (Lexer):** "이 단어는 영어 사전에 있는가?" (예: `if`, `(`, `score`, `)`)
    
- **구문 분석 (Parser):** "이 단어들의 순서가 '주어 + 동사 + 목적어' 같은 영어 문법에 맞는가?" (예: `if (...) { ... }`)
    

### 🎯 구문 분석의 목표

- **입력 (Input):** 어휘 분석기가 넘긴 토큰의 스트림
    
    - `[IF, '(', ID(score), GT, NUM(10), ')', '{', ID(result), '=', NUM(1), ';', '}']`
        
- **출력 (Output):** 코드의 문법 구조를 계층적으로 표현하는 트리. 이것을 **추상 구문 트리 (Abstract Syntax Tree, AST)**라고 부릅니다.
    
- **핵심 역할:** 토큰의 선형적인 순서(Stream)를 **계층적인 구조(Tree)**로 변환합니다.
    

---

### 1. 📜 핵심 이론 1: 문맥 자유 문법 (Context-Free Grammar, CFG)

어휘 분석 단계에서 정규 표현식(Regular Expression)으로 토큰의 '패턴'을 정의했다면, 구문 분석 단계에서는 **문맥 자유 문법(CFG)**으로 문장의 '구조'를 정의합니다.

정규 표현식은 '중첩'이나 '짝맞춤' (예: 괄호)을 표현할 수 없었습니다. CFG는 **재귀(Recursion)**를 이용해 이 문제를 해결합니다.

**예시: 간단한 IF 문법을 CFG로 정의하기**

- `Statement` (문장)
    
- `Expression` (표현식)
    

```
Statement  ->  IfStatement               // 문장은 If문이 될 수 있다
           |   AssignmentStatement       // 또는 대입문이 될 수 있다
           |   ...

IfStatement -> 'if' '(' Expression ')' '{' Statement '}'  // If문의 구조

AssignmentStatement -> ID '=' Expression ';'             // 대입문의 구조
```

CFG의 핵심은 `IfStatement` 정의 안에 `Statement`가 다시 등장하는 **'재귀'**입니다. 이 재귀 덕분에 `if`문 안에 또 다른 `if`문이 중첩되는 복잡한 구조를 완벽하게 기술할 수 있습니다.

---

### 2. 🌳 핵심 산출물: 파스 트리(Parse Tree)와 AST

파서가 CFG 규칙을 바탕으로 토큰 스트림을 분석하면 '트리'가 만들어집니다.

- **파스 트리 (Parse Tree / Concrete Syntax Tree):**
    
    - 문법 규칙(CFG)을 _그대로_ 반영한 트리입니다.
        
    - 괄호 `(`, `)`, 세미콜론 `;` 등 문법에 필요하지만 실제 의미와는 관련이 적은 모든 토큰이 포함됩니다.
        
    - 매우 크고 불필요한 정보가 많습니다.
        
- **추상 구문 트리 (Abstract Syntax Tree, AST):**
    
    - 파스 트리에서 불필요한 정보(괄호 등)를 제거하고, 코드의 **'핵심 의미 구조'** 만을 남긴 트리입니다.
        
    - 컴파일러의 '의미 분석', '최적화' 단계는 모두 이 AST를 기반으로 작동합니다.
        

**예시: `if (x > 10)`**

- **파스 트리:** `IfStatement` -> `if` -> `(` -> `Expression` -> ...
    
- **AST:** `IfNode` 하나에 `Condition` 자식(`> Node`)과 `Body` 자식(`{ ... } Node`)이 바로 연결됩니다.
    

---

### 3. 🥞 핵심 이론 2: 푸시다운 오토마타 (Pushdown Automaton, PDA)

어휘 분석기(Lexer)가 '유한 오토마타(FA)'라는 기계로 동작했다면, 구문 분석기(Parser)는 **'푸시다운 오토마타(PDA)'**라는 이론적 기계로 동작합니다.

**PDA = 유한 오토마타(FA) + 스택(Stack)**

유한 오토마타는 메모리가 없어서 괄호의 짝을 맞출 수 없었습니다. PDA는 **스택(Stack)**이라는 무한한 메모리를 가집니다.

- **작동 원리 (비유):**
    
    1. `{` (열린 중괄호) 토큰을 만나면, 스택에 `PUSH` 합니다.
        
    2. `}` (닫힌 중괄호) 토큰을 만나면, 스택에서 `POP` 합니다.
        
    3. 모든 코드를 다 읽었을 때 스택이 비어있으면, 괄호의 짝이 맞다고 판단합니다.
        

실제 파서는 이 원리를 훨씬 정교하게 사용하여 중첩된 문법 구조 전체를 검증합니다.

---

### 4. 🛠 실제 구현: 파서 생성기 (Parser Generator)

렉서와 마찬가지로, 이 복잡한 PDA/파싱 알고리즘을 직접 코딩하는 것은 매우 어렵습니다. 따라서 **파서 생성기** 도구를 사용합니다.

- **대표적인 도구:** `Yacc` (C언어용), `Bison` (Yacc의 개선판), `ANTLR` (Java, C# 등)
    

**작동 방식:**

1. **개발자:** `.y` 파일(Yacc 파일)에 **CFG 문법 규칙**(위에서 본 예시)과, 해당 규칙이 완성될 때 실행할 코드(AST 노드 생성)를 정의합니다.
    
2. **`Yacc` 도구:** 이 `.y` 파일을 읽어서, 이 문법을 인식하는 **파서(PDA)를 C 코드로 자동 생성**해줍니다.
    
3. **컴파일러:** 렉서가 생성한 C 코드와 파서가 생성한 C 코드를 함께 컴파일하여 컴파일러의 '프론트엔드'를 완성합니다.
    

### 요약 및 다음 단계

구문 분석기는 토큰들을 조립하여 프로그램의 뼈대인 **AST**를 만듭니다. 이로써 컴파일러는 드디어 "코드가 문법에 맞는지"를 넘어서, "이 코드가 **무슨 의미인지**"를 분석할 준비가 되었습니다.

AST가 완성되었으니, 이제 이 트리를 순회하며 코드가 '의미상' 올바른지 검사하는 **의미 분석 (Semantic Analysis)** 단계로 넘어갈 차례입니다.
