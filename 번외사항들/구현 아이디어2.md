좋아. 아래 4개 룰에 대해 **어떤 엔진(린터/AST·CFG/CSA·PP)으로 어떻게 구현할지**를, 정확도·오탐률·개발 난이도 관점에서 **권장 방안(우선순위 포함)**으로 정리했어.  
(전제: Clang 19.x, 우리가 이미 만든 `Guard/Init` 유틸을 재사용해 **빠르게 제품화**하는 목표)

---

## 개요 요약 (권장 엔진)

| 룰                                                               | 핵심 요구               | 1차 권장 엔진                                       | 이유/핵심 포인트                                                                      |
| --------------------------------------------------------------- | ------------------- | ---------------------------------------------- | ------------------------------------------------------------------------------ |
| **다)** const 포인터로 읽는 인자에 **미초기화 변수의 주소** 전달 금지                  | **로컬 경로 내 초기화 여부**  | **린터(AST+CFG, 경로 민감: 함수 내부 한정)**               | 호출 직전까지의 **Definite-Init**만 보면 충분. 외부 함수 본문 몰라도, `const T*`는 “읽을 수 있다” 가정이 안전. |
| **라)** 표현 불가능한 타입으로의 변환 금지(협의상 **Narrowing/Out-of-range cast**) | **값이 대상 타입에 들어가는지** | **린터(AST)** → 상수·즉시 검증 / (옵션) **AST+경량 범위 전파** | 상수식/즉시식은 **정확**. 변수는 “가드가 보장하는 범위”만 허용(간단한 범위 전파). 일반 케이스는 CSA로 확장 가능.         |
| **가)** 역참조 전 **NULL 체크 필수**                                     | **경로 상 보장된 비-NULL** | **린터(AST+CFG, 가드 인식)**                         | 우리가 만든 **가드 탐지(If/Assert, 지배 조건)**를 재사용. 함수 내부에서 충분히 정확. 함수 간은 옵션.             |
| **나)** 변수로 나누기 전 **0 체크 필수**                                    | **경로 상 보장된 분모≠0**   | **린터(AST+CFG, 가드 인식)**                         | (가)와 동일한 가드 인식 틀을 재사용. 0 비교/`if (d)`/`assert(d)`/전처리 가드까지 포괄.                  |

> 공통 전략: **하나의 소형 데이터흐름 유틸**(Init/Null/Range)을 만들어 **여러 체크가 공유**하도록 하고, **보수적 기준**(+프로젝트 옵션)을 둬서 오탐을 줄이고 도입을 쉽게 한다.

## (다) 값이 초기화되지 않은 변수의 포인터를 함수의 Read-only(const)로 사용 금지

### 무엇을 잡나

```c
void use(const int* p);
void g() {
  int x;            // 미초기화
  use(&x);          // ❌ const로 읽힐 수 있는 주소를 전달
}
```

### 권장 구현 (린터 / AST+CFG, 함수 내부 한정)

1. **매칭**
    
    - `callExpr(callee(functionDecl().bind("callee")), hasArgument(I, unaryOperator(hasOperatorName("&"), hasUnaryOperand(declRefExpr(to(varDecl().bind("v"))))))).bind("call")`
        
    - 또는 `array-to-pointer decay`도 커버(예: `use(arr)`)
        
2. **“읽기 의도” 판단**
    
    - 인자 위치 `I`의 파라미터 타입 `P`가 `const T*`(또는 `pointer-to-const`)이면 **읽을 수 있음**으로 가정.
        
    - (옵션) 레퍼런스 `const T&`도 포함.
        
3. **초기화 상태 데이터흐름 (함수 내부 CFG)**
    
    - **격자**: `Uninit / Init / Unknown` (경로-민감 머지 시 `Uninit`이 하나라도 섞이면 `MaybeUninit`로 진단)
        
    - 소스: `int x = 0;` → Init, `x = ...;` → Init, `memset(&x,0,sizeof x)`(옵션 패턴) → Init
        
    - 싱크: 호출 시점에 `&x` 전달 → `x`가 **모든 도달 경로에서 Init**이 아니면 경고
        
    - **수정 감지**: 가드와 사용 사이에 `x`를 변경하는 문장이 있으면 상태 갱신
        
4. **거짓 양성 완화**
    
    - **출력 전용 파라미터** (`T*` 비-const) 는 **경고 제외** (callee가 채우는 용도)
        
    - `use(&x)` 앞에 `x`에 대한 명확 초기화가 있는 패턴(대입/초기화/`memset`)은 허용
        
    - 매크로 확장은 기본 제외(옵션)
        
5. **옵션**
    
    - `AssumeConstPointerReads=true` (기본): `const T*`는 읽는다고 가정
        
    - `HonorStdLibInitPatterns=true`: `memset(&x,0,sizeof x)` 등 허용 패턴 화이트리스트
        
    - `OnlyMainFile`,`IgnoreSystemHeaders`,`IgnoreMacros`
        

> **왜 린터가 적합?** Callee 본문 없이도 **“읽을 가능성”**만으로 안전하게 경고할 수 있고, 함수 내부의 Definite-Init만 보면 충분히 정확함.

## (라) Object 저장 값을 표현할 수 없는 데이터로의 형 변환 금지 (Narrowing/Out-of-range cast)

### 무엇을 잡나

```c
short s = 40000;          // ❌ 범위 초과 상수 대입
int x = -1; unsigned char c = x;  // ❌ 음수 → 부호 없는 좁은 타입
int i = 300; char ch = (char)i;   // ⚠️ 경계 불명(가드 없으면 경고)
```

### 권장 구현 (1차: 린터/AST, 2차: 가벼운 범위 전파)

1. **매칭 대상**
    
    - **암시적/명시적 변환** 모두:
        
        - `varDecl(hasInitializer(expr))` (초기화 대입)
            
        - `binaryOperator(hasOperatorName("="))` (대입)
            
        - `returnStmt(hasReturnValue(expr))` (반환)
            
        - `cStyleCastExpr`, `staticCastExpr`, `implicitCastExpr(CK_IntegralCast/CK_FloatingToIntegral/CK_IntegralToFloating ...)`
            
2. **상수 즉시 검증 (정확)**
    
    - RHS가 정수/부동 **상수식**이면, `APSInt/APFloat`로 평가 → 대상 타입 `QualType`의 표현범위에 **정확히 포함되는지** 판정
        
        - 정수→정수: 폭/부호로 범위 비교
            
        - 부동→정수: `APFloat::convertToInteger`로 오버플로/NaN/Inf 확인
            
        - 정수→부동: 정밀 손실만으로 경고할지 옵션화
            
3. **변수에 대한 “즉시 가드” 인식 (경량 범위 전파)**
    
    - **동일 블록·지배 조건**에서의 비교를 인식:
        
        - `0 <= i && i <= 255` → `unsigned char`로 안전
            
        - `i < 0` 차단 등
            
    - 우리가 Null/Zero 가드에서 쓰던 **지배 조건 유틸**을 재사용: `conditionImpliesRange(var, [L,U])`
        
    - 가드 이후 **사용 전 수정**(재대입/증감) 감지
        
4. **정책**
    
    - **상수 초과/부적합**: **무조건 경고**
        
    - **변수**: **가드 없으면 경고**(옵션) / 가드로 상계가 명확하면 허용
        
5. **옵션**
    
    - `WarnOnPotentialNarrowing=true` (변수·가드 미확인시 경고)
        
    - `WarnFloatToIntAlways=true` (부동→정수는 항상 경고)
        
    - `RespectGuards=true` (기본 on)
        

> 1차만으로도 **상수 기반** 오류는 모두 정확히 잡힘. 2차를 추가하면 실용성이 크게 올라가지만, 오탐 억제를 위해 “지배 조건+수정 감지”만 적용하는 **얕은 데이터흐름**을 추천.

## (가) 포인터는 참조 전에 NULL 여부 확인

### 무엇을 잡나

```c
void f(int* p) {
  *p = 1;           // ❌ 가드 없음
}
void g(int* p){
  if (!p) return;
  *p = 1;           // ✅ 가드 후 사용
}
```

### 권장 구현 (린터 / AST+CFG, 가드 인식)

1. **매칭**
    
    - 역참조 전부: `unaryOperator("*")`, `memberExpr(isArrow())` (`p->m`), `arraySubscriptExpr`(베이스 포인터), `*(p+k)`
        
2. **가드 인식(지배 조건)**
    
    - `if(p)`, `if(p != NULL)`, `if(NULL != p)`, `assert(p)`(옵션), `if (!p) return;` 패턴을 **도달 경로** 기준으로 검사
        
    - 우리가 이전에 만든 `findNearestEnclosingIf` + `conditionImpliesNonNull(var)` 재사용
        
3. **수정 감지**
    
    - 가드 이후 **사용까지 사이**에서 `p` 재대입/함수콜(알 수 없는 수정) → 보수적으로 가드 무효
        
4. **정책**
    
    - **가드 없으면 경고**
        
    - `std::unique_ptr`/레퍼런스는 제외(옵션)
        
5. **옵션**
    
    - `RecognizeAssertLike={"assert","MY_ASSERT"}` 화이트리스트
        
    - `AssumeCtorNonNull=false` (생성 직후라도 가드 필수로 볼지)
        
    - `OnlyMainFile`, `IgnoreSystemHeaders`, `IgnoreMacros`
        

> 로컬 함수 범위에서 **충분히 정확**. 함수 경계를 넘는 추적은 CSA로 확장.

## (나) 변수로 나누기 전 0 체크

### 무엇을 잡나

```c
int div(int a, int d) {
  return a / d;         // ❌ d가 0일 수 있음
}
int ok(int a, int d){
  if (d == 0) return -1;
  return a / d;         // ✅ 가드 존재
}
```

### 권장 구현 (린터 / AST+CFG)

1. **매칭**
    
    - `binaryOperator(hasAnyOperatorName("/", "%"), hasRHS(expr().bind("den")))`
        
2. **즉시 상수 0**
    
    - RHS가 상수식 0 → 즉시 경고
        
3. **가드 인식**
    
    - `if(d != 0)`, `if(d)`, `assert(d)`(옵션) 등 **지배 조건** 확인
        
    - `!= 0`, `> 0`, `< 0`은 **0 아님**으로 인정(부호는 상관없음)
        
4. **수정 감지**
    
    - 가드 후 사용 전 `d` 변경되면 가드 무효
        
5. **옵션**
    
    - `RecognizeAssertLike`, `OnlyMainFile`, `Ignore...`
        

> (가)와 **같은 가드 엔진**을 그대로 재사용하니, 구현·성능·정확성 모두 **좋은 비용대비효과**를 보임.

## 공통 인프라 (재사용 추천)

한 번 만들어 여러 체크에서 공유:

- `bool conditionImplies(var, PredicateKind, Value/Range, inThen)`  
    (예: `!= 0`, `== NULL`, `0<=x && x<10`)
    
- `bool modifiedBeforeUse(Subtree, var)`  
    (가드~사용 사이 재할당/증감/복합대입 감지)
    
- `std::optional<long long> evalIntegerConstant(expr)`
    
- `const IfStmt* findNearestEnclosingIf(UseSite, &inThen)` (+ while/for 조건 지배 인식)
    
- `RecognizeAssertLike`(매크로명 화이트리스트)
    

> 이미 `NullPointerArithmeticCheck`에서 만든 틀을 **유틸로 분리**하면, 4개 룰이 **같은 논리**를 공유하면서도 **빠르게 제품화** 가능.

## 우선 적용 순서(추천)

1. **나(0으로 나누기) + 가(NULL 가드)**  
    → 같은 가드 엔진을 공유, 즉효성/가독성/안정성 최고.
    
2. **다(미초기화 & const 포인터 인자)**  
    → 우리 Definite-Init 틀 재사용. 미탐 줄이기 쉬움.
    
3. **라(표현 불가능 변환)**  
    → 1차(상수/즉시식)부터 묶고, 점진적으로 “가드 기반 범위 전파”를 추가.

## 예시 진단 메시지(한글)

- **다)** `'%0'는 초기화되지 않았습니다. 읽기 전용 포인터 인자('%1')로 주소를 전달하기 전에 값을 초기화하십시오.`
    
- **라)** `값 %0 은 타입 '%1'로 안전하게 표현할 수 없습니다(범위 초과/손실).`
    
- **가)** `포인터 '%0'는 역참조 전에 NULL 여부를 확인해야 합니다. 가드(if/assert)를 추가하십시오.`
    
- **나)** `분모 '%0'는 0이 아님이 경로상 보장되어야 합니다. 'if (%0 != 0)' 등의 가드를 추가하십시오.`
