**LLVM**은 특정 프로그래밍 언어나 아키텍처에 종속되지 않는, 재사용 가능한 **컴파일러 기반 구조**(Compiler Infrastructure)입니다. 이름에 '가상 머신(Virtual Machine)'이 있지만, 지금은 자바 가상 머신(JVM)과는 다른, 컴파일러를 만드는 데 필요한 모듈화된 부품들의 집합체(a collection of modular and reusable compiler and toolchain technologies)라고 이해하는 것이 정확합니다.



전통적인 컴파일러는 크게 3단계로 나뉩니다.

1. **프론트엔드 (Frontend):** 개발자가 작성한 소스 코드(C++, Swift 등)를 해석해서 컴퓨터가 이해하기 쉬운 중간 형태(Intermediate Representation, IR)로 변환합니다. 언어 종속적인 부분입니다.
    
2. **옵티마이저 (Optimizer):** 중간 형태의 코드를 분석하여 더 빠르고 효율적으로 동작하도록 최적화합니다.
    
3. **백엔드 (Backend):** 최적화된 중간 코드를 실제 기계가 실행할 수 있는 기계어(x86, ARM 등)로 변환합니다. 아키텍처 종속적인 부분입니다.

## 그럼 왜 탄생했는가?

초기 애플(Apple)은 GCC를 공식 컴파일러로 사용했습니다. 하지만 애플이 Objective-C 언어를 발전시키려 할 때마다 GCC 커뮤니티와 마찰이 있었고, 이로 인해 애플이 사용하는 GCC 버전은 공식 버전보다 뒤처지기 시작했습니다.

또한, GCC는 코드 전체가 너무 긴밀하게 얽혀있어(일체형 구조), 애플이 원했던 현대적인 개발 환경(IDE) 지원처럼 특정 기능만 떼어내어 사용하기가 매우 어려웠습니다.

이러한 문제들 때문에 애플은 더 효율적이고, 모듈화되었으며, 자유로운 대안을 찾기 시작했습니다. 결국 컴파일러 전문가인 **크리스 래트너**(Chris Lattner)를 영입하여 LLVM 프로젝트를 이끌게 했고, 이것이 LLVM의 본격적인 시작이었습니다.


>LLVM은 크리스 래트너가 애플에 입사하기 전인 **2000년부터 일리노이 대학교(UIUC)에서 비크람 애드브 교수와 함께 시작한 연구 프로젝트**였습니다.

- 해당 예시의 GCC는 간단하게 설명하기 위한 예시일 뿐이지 GCC 또한 GIMPLE이라는 IR이 존재하였습니다, 다만 높은 코드 결합도, 모듈화의 부재, Objective-C 지원에 대한 불협화음등에 대한 단점으로 llvm개발의 동기가 되었습니다.

## 작동 순서

### 1. 프론트엔드 (Frontend): 소스 코드를 이해하고 번역하는 단계

프론트엔드는 사람이 작성한 소스 코드를 컴퓨터가 분석할 수 있는 구조적인 형태로 만드는 것을 목표로 합니다. 마치 외국어 문장을 번역하기 위해 단어를 나누고 문법 구조를 파악하는 것과 같습니다.

1. **전처리 (Preprocessing):** `#include`나 `#define` 같은 전처리기 지시문을 처리합니다. 헤더 파일을 소스 코드에 삽입하고 매크로를 실제 코드로 치환하여 순수한 소스 코드 파일로 만듭니다.
    
2. **어휘 분석** (Lexical Analysis): 전처리된 코드를 의미 있는 최소 단위인 **토큰**(Token)으로 분해합니다. 예를 들어, `int a = 10;`은 `int`, `a`, `=`, `10`, `;` 다섯 개의 토큰으로 나뉩니다.
    
3. **구문 분석 (Syntax Analysis):** 토큰들의 나열을 언어의 문법 규칙에 따라 분석하여 **추상 구문 트리(Abstract Syntax Tree, AST)** 라는 나무 형태의 자료구조로 만듭니다. 이 과정에서 문법 오류(예: 괄호 불일치)를 잡아냅니다.
    
4. **의미 분석 (Semantic Analysis):** AST를 기반으로 코드의 의미가 유효한지 검사합니다. 예를 들어, 선언되지 않은 변수를 사용하거나, 정수형 변수에 문자열을 할당하는 등의 타입 오류를 이 단계에서 확인합니다.
    
5. **중간 표현 (IR) 생성:** 의미 분석까지 완료된 AST를 기반으로 다음 단계인 옵티마이저가 사용할 수 있는 표준화된 중간 형태(LLVM IR)로 변환합니다.

### 2. 옵티마이저 (Optimizer): 코드를 더 효율적으로 다듬는 단계

옵티마이저는 프론트엔드에서 생성된 IR을 입력받아, 코드의 의미는 바꾸지 않으면서 더 빠르고 효율적으로 동작하도록 개선하는 작업을 수행합니다. 이 단계는 수많은 **최적화 패스(Pass)** 들이 순차적으로 또는 반복적으로 적용되는 과정입니다.

1. **분석 패스 (Analysis Passes):** 먼저 코드의 구조와 데이터 흐름을 분석합니다. 예를 들어, 어떤 변수가 루프 안에서 변하지 않는지, 특정 코드가 실제로 실행될 가능성이 있는지 등을 파악합니다.
    
2. **변환 패스 (Transform Passes):** 분석된 정보를 바탕으로 코드를 직접 수정합니다.
    
    - **죽은 코드 제거:** 절대 실행되지 않는 코드를 삭제합니다.
        
    - **루프 불변 코드 이동:** 루프 내에서 결과가 변하지 않는 계산을 루프 밖으로 빼내어 반복적인 연산을 줄입니다.
        
    - **함수 인라이닝:** 간단한 함수 호출을 함수 본문 코드로 대체하여 호출 오버헤드를 줄입니다.
        
    - 이 외에도 수십, 수백 개의 다양한 최적화 패스가 IR을 계속해서 개선합니다.

### 3. 백엔드 (Backend): 실제 기계가 알아듣는 언어로 번역하는 단계

백엔드는 최적화된 IR을 특정 CPU 아키텍처(x86, ARM 등)에서 실행할 수 있는 최종 기계어로 변환하는 역할을 합니다.

1. **명령어 선택 (Instruction Selection):** 플랫폼에 독립적인 IR 명령어를 실제 목표 CPU의 어셈블리 명령어로 변환합니다. 예를 들어, IR의 `add` 명령어를 x86의 `ADD` 명령어로 대응시키는 과정입니다.
    
2. **레지스터 할당 (Register Allocation):** 프로그램의 수많은 변수들을 CPU의 한정된 개수의 물리적 레지스터에 효율적으로 할당합니다. 레지스터가 부족할 경우, 일부 데이터를 메모리(스택)에 저장했다가 다시 불러오는 코드를 추가합니다.
    
3. **명령어 스케줄링 (Instruction Scheduling):** CPU의 파이프라인 성능을 극대화하기 위해 어셈블리 명령어의 순서를 재배치합니다. 예를 들어, 특정 명령어의 계산 결과가 나올 때까지 기다리는 동안 다른 명령어를 먼저 실행하도록 순서를 조정합니다.
    
4. **코드 생성 (Code Generation):** 최종적으로 어셈블리 코드를 생성하고, 이를 목적 파일(`.o`)이나 라이브러리 파일 등 실행 가능한 형태의 바이너리 코드로 만듭니다.
