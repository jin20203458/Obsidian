
**전처리기(Preprocessor)**는 Clang 프론트엔드에서 가장 먼저 실행되는 단계입니다. 주된 역할은 실제 파싱(Parsing)이 시작되기 전에, `#` 기호로 시작하는 특별한 지시문들을 처리하여 소스 코드를 깨끗하게 정리하는 것입니다. 본격적인 요리(파싱)에 들어가기 전 '재료 손질**' 단계라고 생각할 수 있습니다.


전처리기의 주요 작업은 다음과 같습니다.

- **파일 포함 (`#include`):** 헤더 파일의 내용을 소스 파일에 삽입합니다.
    
- **매크로 확장 (`#define`):** 코드에 정의된 매크로를 실제 값이나 코드로 치환합니다.
    
- **조건부 컴파일 (`#if`, `#ifdef` 등):** 특정 조건에 따라 코드 블록을 컴파일에 포함하거나 제외합니다.
    

일반적으로 컴파일러에게 전처리 과정은 일방통행과 같습니다. 일단 실행되어 코드를 변환하고 나면, 원래의 지시문 정보는 사라지죠. 하지만 사용자 정의 체커에게는 이 사라지는 정보가 매우 중요할 수 있습니다. 바로 이 지점에서 `PPCallbacks`가 활약합니다.

> AST는 이미 해당 과정을 끝마친 상태로 제작되기때문에, 전처리기가 이미 작업한 내용에 대한 정보를 알수가 없습니다. 그렇기에 해당 관련 검사 예를들어 사) 전처리에 사용되는 매크로는 사용 전에 반드시 정의되어야 한다. 같은 룰을 검사하는 체커를 제작하기 위해서는 해당과정이 필수 불가결합니다.

---

### PPCallbacks: 전처리 과정을 엿듣는 탐지기 

`PPCallbacks`(Preprocessor Callbacks)는 Clang이 제공하는 기능으로, 도구가 **전처리기에 훅(hook)을 걸어 특정 이벤트가 발생할 때마다 알림(notification)을 받을 수 있게** 해줍니다. 마치 전처리기가 특정 지시문을 만날 때마다 "지금 `#include` 만났음!" 하고 알려주는 경보 시스템을 설치하는 것과 같습니다.

이 기능이 필수적인 이유는, AST가 만들어지고 나면 매크로, 어떤 헤더가 포함되었는지, 조건부 컴파일 블록 같은 **정보들이 대부분 사라지기 때문**입니다. 만약 체커가 매크로가 어떻게 정의되었는지, 또는 특정 헤더가 포함되었는지 등을 분석해야 한다면, AST 단계는 너무 늦습니다. 반드시 `PPCallbacks`를 사용해야 합니다.

주요 `PPCallback` 이벤트는 다음과 같습니다.

- `InclusionDirective`: 전처리기가 `#include` 지시문을 만났을 때 호출됩니다. 어떤 파일이 포함되는지 알 수 있습니다.
    
- `MacroDefined` / `MacroUndefined`: `#define` 또는 `#undef` 지시문이 처리될 때 호출됩니다.
    
- `MacroExpands`: 코드에서 매크로가 사용되어 확장될 때 호출됩니다. 문제 있는 매크로 사용법을 찾는 데 유용합니다.
    
- `If`, `Ifdef`, `Ifndef`: 조건부 컴파일 블록이 시작될 때 호출됩니다.
    

`PPCallbacks` 클래스를 상속받는 체커를 만들고 이 메서드들을 오버라이드(재정의)하면, 전처리기가 해당 지시문을 처리하는 바로 그 순간에 원하는 분석 로직을 실행할 수 있습니다.

```cpp
// PPCallbacks를 사용하는 체커의 개념 코드
class MyChecker : public ClangTidyCheck,
                  public PPCallbacks {
public:
  // ... 다른 체커 설정 ...

  // 이 메서드를 통해 전처리기에 콜백을 등록합니다.
  void registerPPCallbacks(/*...인자들...*/) override;

  // 이 메서드는 #include 지시문을 만날 때마다 호출됩니다.
  void InclusionDirective(SourceLocation HashLoc,
                          const Token &IncludeTok,
                          // ...기타 인자들...
                          const Module *Imported) override {
    // 포함된 파일 이름을 검사하는 등의 분석 로지 구현
    diag(HashLoc, "#include 지시문을 여기서 발견했습니다.");
  }
};
```
> 단 이때 호출되는 체커는 AST 기반이 아니기 때문에 우리가 AST 기반 체커에서 사용할수 있는 많은 기법들 (재귀적 순회, Vist 패턴등)을 사용할수 없기때문에 Laxer 혹은 자체적으로 문자열 처리 파서를 제작하여  문자열 자체를 분석하는 방식을 많이 사용합니다. 

[[MacroDefinedBeforeUseCheck]] 등이 대표적인 예시입니다.
